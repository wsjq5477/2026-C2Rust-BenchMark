# FlashDB Rust 架构

## 目的

本文件记录后续迁移检查点使用的 Rust 架构原则。架构必须从当前 FlashDB C 输入、模型产物和测试事实中推导，不得在 knowledge 中预置固定函数、固定类型或固定文件清单。

## 模块推导

- Rust 项目的根目录由比赛约束确定为 `flashDB_rust`。
- 具体源码文件、测试文件和模块边界必须由 `rust_api_design.json` 中的模块设计决定。
- 模块划分应对应 C API 分组、数据结构职责、存储抽象和测试覆盖需求。
- 如果当前输入暴露扩展能力或额外子系统，设计产物应记录是否纳入模块、保留为扩展证据，或明确标为 unmapped。

## 类型推导

- Rust public types 必须由当前 C public API、结构体/枚举事实、错误返回语义和测试使用方式推导。
- 不得因为 knowledge 文档中的示例名而创建固定类型；类型名称和可见性以 `rust_api_design.json` 为准。
- 错误模型应覆盖当前 C API 可观察到的失败模式，并用 Rust 的类型系统表达。
- 存储抽象应来自 C 侧 flash/partition/sector 语义和测试中的持久化行为，而不是来自预设文件模板。

## 语义原则

- 优先使用 safe Rust。
- `HashMap` 只作为运行时索引使用。
- 保留 C 输入和测试事实中可观察到的追加、删除、重载、清理、迭代和状态变更语义。
- 删除、回收、重启恢复等行为必须由 C 测试事实和模型证据支持，不能按预置答案硬编码。
- reload 类行为必须基于 storage 内容重建状态，不能只依赖内存索引。
- 清理或回收逻辑必须保留测试事实要求的有效数据。

第一个 framework checkpoint 不生成 Rust 源码。
