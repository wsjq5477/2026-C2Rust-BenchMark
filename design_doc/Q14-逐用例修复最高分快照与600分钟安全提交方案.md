# Q14 - 逐用例修复、最高分快照与 600 分钟安全提交方案

> 状态：已实施；2026-07-16 已接入逐任务 repair queue、无 Git submission snapshot、540 分钟 watchdog freeze、deadline FINAL gate 与回归测试
>
> 日期：2026-07-16
>
> 范围：`02_02` 的 C-Cross/Rust 测试修复调度、无 Git 最优状态缓存、540 分钟冻结恢复、最终报告与 gate。不得修改平台 C 输入，不预置固定用例答案，不依赖本地 Git 仓库。
>
> 已确认约束：竞赛运行硬上限为 600 分钟，超时计 0 分；正常修复不使用时间动态调度；到运行第 540 分钟必须停止修改，恢复当前最高分且同分时最新的完整可提交快照，并尽快结束任务。

## 1. 结论

当前工作流以全局 repair 次数限制整个阶段，容易被一个困难场景占住流程，也无法保证已经获得的较高通过率不会被后续修改破坏。现有 `c_cross_validate.py` 虽然已有不依赖 Git 的 `src/**` 最优快照，但它只按 C-Cross 结果保存 Rust 生产源码，不能保护 Rust tests、Cargo 文件、mapping 和用例一致性结果，也不能在竞赛截止前无条件恢复。

本次设计将流程改为三个互相配合、但职责独立的机制：

```text
失败任务工作队列
    └─ 逐个修复，难例暂存，先覆盖所有可修场景

runtime-best
    └─ 继续承担 C-Cross 修复期间的源码回归恢复

submission-best + contest watchdog
    └─ 保存评分意义上的完整最优提交
       到 540 分钟无条件恢复、冻结并提交
```

核心停止条件不再是“整个项目修了三轮或八轮”，而是：

1. 正常阶段中，失败工作队列是否还有未处理或可继续处理的任务；
2. 单个任务是否已经完成两遍有界修复且仍无进展；
3. 竞赛运行是否到达第 540 分钟冻结点。

第 540 分钟是唯一的时间安全阀，不用于平时决定哪个用例值得修复，也不根据 token、API budget 或模型预算动态缩短工作。

## 2. 当前工程事实与缺口

### 2.1 已有的无 Git C-Cross 回滚

`02_02/work/tools/c_cross_validate.py` 当前已经提供：

- 以复制文件的方式缓存 `flashDB_rust/src/**`；
- 为缓存文件保存 SHA-256 manifest；
- 按 C-Cross passed/not-run/failed 和失败推进深度比较 best-known；
- 通过 `--restore-best` 恢复文件；
- 恢复时删除快照中不存在的多余 `src/**` 文件；
- 写入 `logs/trace/c-cross/restore-best.json` 作为恢复证据。

该机制不依赖 Git，方向正确，应继续作为 `runtime-best` 使用。

### 2.2 当前回滚不能承担最终提交保护

现有机制存在以下限制：

1. 快照范围只有 `src/**`，不包含 `tests/**`、Cargo 文件和测试 mapping；
2. best score 只反映 C-Cross，不反映评分平台优先使用的用例一致性与真实 Rust 测试结果；
3. `--restore-best` 只允许紧跟已记录 regression 使用，不能在第 540 分钟无条件恢复；
4. 没有保存与完整提交文件绑定的验证证据；
5. 恢复后仍可能被正在运行的 primary/subagent 再次覆盖；
6. 没有独立于 Agent 上下文的冻结信号和看门狗。

### 2.3 评分口径

当前评分 Skill 明确：

- Cargo build 失败时最终通过率为 0；
- 用例一致性是 Cargo 测试结果的前置门禁；
- 缺失、错配或逻辑不一致的 Rust 测试，即使 `cargo test` 显示通过，对应标准用例仍为失败；
- 得分主要由动态标准用例的真实 passed 数量决定；
- timeout 和 not-run 不得计为 passed。

因此，最终快照不能只追求 C-Cross passed 数，也不能把空测试或自证 mapping 计入最高分。

### 2.4 竞赛入口边界

评测平台通过加载 `INSTRUCTION.md` 驱动 Agent，没有由本工程直接控制的统一 OpenCode launcher。因此：

- 工程无法精确读取平台官方任务创建时刻；
- 工程不能依赖 Agent 自己在对话中估算时间；
- 工程不应在第 540 分钟通过发送 SIGTERM 杀死 OpenCode，异常退出可能仍被判超时或失败；
- 可以把看门狗作为 `INSTRUCTION.md` 的第一条确定性命令启动，并用 60 分钟安全余量吸收平台启动到第一条命令之间的微小偏差。

### 2.5 旧方案硬门限审计

逐任务优化不能直接叠加到当前实现。当前代码和合同仍存在以下硬门限：

| 旧门限/判断 | 当前位置 | 与本方案的冲突 |
|---|---|---|
| IMPLEMENT 全局最多 1 次 repair | `core_impl_audit.py: MAX_IMPLEMENTATION_REPAIR_ATTEMPTS = 1`，并由 gate/report/INSTRUCTION 解释为耗尽后 `failed_final` | 一个 finding cluster 未完成就会让整个迁移在测试前终止，其他可实现模块没有机会继续 |
| C-Cross 全局最多 8 次 repair | `c_cross_validate.py: MAX_REPAIR_ATTEMPTS = 8`，`gate.py` 和 `report_writer.py` 另写死期望值 8 | 8 次被少数难例消耗后，尚未处理的失败场景直接失去修复机会 |
| Rust 测试全局最多 2 次 repair | `cargo_capture.py: MAX_TEST_REPAIR_ATTEMPTS = 2` | 任意 tests/mapping 文件变化都消耗同一个全局计数，无法逐用例轮转 |
| 单次测试 repair 最多改变 4 个 tracked 文件 | `cargo_capture.py: MAX_TEST_REPAIR_CHANGED_FILES = 4` | 一个合法任务同时涉及 test、helper、mapping 和经授权的生产源码时可能被机械拒绝 |
| failure cluster 最多 3 个 scenario | `c_cross_validate.py: MAX_CLUSTER_SCENARIOS = 3` | 同一根因动态影响 4 个以上场景时会被人为拆开，重复诊断和修改 |
| local repair 最多 3 个 Rust 源文件、architecture repair 最多 6 个 | `c_cross_validate.py`、`rust-implementer.md` | 属于任务范围保护，不是全局终止条件；可以保留，但不能据此直接形成 `failed_final` |
| 同一 cluster 连续 2 次 local 无进展后才允许 architecture | `LOCAL_STALLS_BEFORE_ARCHITECTURE = 2` | 与“单任务失败后让出队列”方向兼容，但状态必须记录到该 task，不能读取全局最后两条 repair 推断 |
| 每次定向 C-Cross repair 后必须立即全量 confirmation | `INSTRUCTION.md`、orchestrator | 与定向验证、suite 回归、每遍结束再全量验证的执行策略冲突，放大运行次数 |
| placeholder 全局未通过时禁止 Cargo | `cargo_capture.validate_placeholder_precondition` | 一个缺失测试会阻止其他已就绪场景取得真实 Cargo 结果，无法先提高可得分用例数 |
| semantic review 必须一次审查全部动态场景 | `test-semantic-reviewer.md` | 单次上下文过大，也无法对刚修好的用例做局部 current review |
| 顶层 semantic review 非 pass 时，其他无 issue 场景也只能 unresolved | `test_consistency_check.py` 的 `review_is_current_and_passed` | 这是直接的按场景计分冲突：一个坏用例会抹掉其他好用例的 consistency passed 数 |
| TEST_AND_REPORT 成功或失败只看“全局全过/全局 2 次耗尽” | `gate.py.evaluate_test_stage`、`report_writer.py` | 无法表达 queued/deferred/exhausted 的逐任务终态 |
| test-migrator 必须按整个 `rust_file` 分组 | `INSTRUCTION.md`、orchestrator | 一个文件包含许多场景时仍会形成大任务，不符合逐失败任务轮转 |

对应的现有回归测试也写死了 1、8、2 等值。开发不能只修改常量；必须同时迁移状态模型、gate、report、文档和测试。

### 2.6 硬门限迁移决定

本方案对旧门限作如下处理：

**删除并替换：**

1. 删除 IMPLEMENT 全局 1 次、C-Cross 全局 8 次、Rust tests 全局 2 次终止预算；
2. `failed_final` 不再由全局 numeric budget 耗尽触发，改为任务队列全部 resolved/exhausted 后的全量确认，或合法的 540 分钟 deadline freeze；
3. 删除 gate/report 对固定 `max_repair_attempts` 数值的要求；
4. 删除 placeholder 全局失败对所有 Cargo 执行的绝对阻塞；
5. 删除 consistency 对顶层 reviewer 全过的逐场景连带依赖；
6. test-migrator 从整文件必需任务改为按 queued scenario/最小共享 helper cluster 调度；
7. 删除“每个定向 repair 后立即全量 confirmation”的合同，改为定向验证、受影响 suite 回归和每遍全量确认。

**保留但改变作用域：**

1. 同一 task 连续 2 次 local 无进展后切换诊断/architecture，作为让出队列和升级策略的依据；
2. local 3 文件、architecture 6 文件暂作为单任务修改范围保护；超过时拆成有依赖顺序的多个 task，不把超限解释为整个项目失败；
3. changed-file 精确 manifest、freshness、禁止 stale evidence、禁止 0 tests 假通过等真实性门禁继续保留；
4. 正常 final 仍要求全量验证，deadline final 仍要求恢复 validated snapshot 并验证 hash/evidence 闭环。

**改为动态约束：**

1. cluster 不再固定最多 3 个 scenario，而是要求同一失败指纹、共同根因或共同修改文件证据；
2. 测试 task 不再固定最多 4 个 tracked 文件，而是必须在 task 创建时声明动态 allowlist，实际变化必须与 allowlist 一致；
3. attempt 数仍记录，但只用于 task 审计、无进展判断和调度公平性，不作为全项目统一预算。

## 3. 设计目标与非目标

### 3.1 目标

1. 不让单个困难用例阻塞其他可修用例。
2. 每个失败任务都至少完成第一遍处理；第一遍难例暂存后继续处理队列。
3. 对通过数有真实提升的可提交状态自动保存，不依赖 Git。
4. 快照必须覆盖评分所需的 Rust 源码、Rust 测试和配置，且具备内容 hash。
5. 快照分数必须来自当前文件对应的真实验证证据，不能使用 stale JSON。
6. buildable 状态始终优先于无法构建的状态，避免恢复后直接得到 0 分。
7. 第 540 分钟无条件停止修改，恢复“最高分、同分时最新”的完整提交。
8. 恢复后禁止继续修改提交文件，并在平台 600 分钟硬截止前正常结束。
9. 不硬编码固定用例数量、固定 case id、KVDB/TSDB 文件名或样例答案。
10. 保留全部未通过用例及原因，不能把 deadline freeze 伪装成全部成功。

### 3.2 非目标

- 不恢复已删除的通用 workflow controller、task packet、worker receipt 或 revision 系统。
- 不使用 Git commit、stash、branch 或 worktree 作为正确性前提。
- 不用时间剩余量选择用例、估算模型能力或动态减少正常修复次数。
- 不把 provisional 的定向通过结果直接当成最终最高分。
- 不在 540 分钟后重新开启耗时修复或完整迁移流程。
- 不由看门狗强杀 OpenCode 进程。
- 不根据模型 token、API quota 或费用决定流程终态。

## 4. 总体流程

```text
INSTRUCTION 第一条命令启动 contest watchdog
        ↓
ANALYZE / IMPLEMENT
        ↓
首次 buildable 状态
        └─ 保存 submission baseline
        ↓
C-Cross 全量 checkpoint
        ↓
构建失败任务工作队列
        ↓
第一遍：逐任务定向修复，难例暂存
        ↓
suite/full 确认并提升 runtime-best
        ↓
迁移并执行 Rust tests / consistency
        ↓
用例级修复工作队列
        ↓
真实评分提升时提升 submission-best
        ↓
第二遍：重新处理暂存和回归任务
        ↓
正常完成
        ├─ success：生成报告并停止 watchdog
        └─ 队列耗尽仍失败：failed_final，生成报告并停止 watchdog

任意阶段到达 540 分钟
        ↓
FREEZE_REQUESTED
        ↓
恢复 submission-best
        ↓
hash 与证据指纹校验
        ↓
SUBMISSION_FROZEN
        ↓
生成 deadline final 报告并立即结束任务
```

## 5. 逐用例失败工作队列

### 5.1 调度单位

对外记录单位始终是动态 `scenario_id`。实际修复任务允许是：

- 单个失败场景；或
- 具有明确共同根因、共同 Rust 文件和共同失败指纹的最小 cluster。

例如多个 GC 场景都由同一地址维护错误导致时，可以作为一个 cluster 修复；但每个 scenario 的状态、通过结果和尝试历史仍分别记录。不得为了减少任务数，把无共同证据的整个 suite 合并为一个大任务。

### 5.2 任务状态

每个动态任务具有以下状态：

```text
queued
active
provisional_pass
confirmed_pass
deferred_no_progress
regressed
exhausted
```

含义如下：

| 状态 | 含义 |
|---|---|
| `queued` | 尚未处理，必须保留在工作队列 |
| `active` | 当前正在诊断或进行一次真实修改 |
| `provisional_pass` | 定向场景已通过，但尚未完成 suite/full 回归检查 |
| `confirmed_pass` | 当前文件版本下通过了要求的回归范围 |
| `deferred_no_progress` | 第一遍连续无进展，暂存后先处理其他任务 |
| `regressed` | 曾通过但被后续修改破坏，需要恢复或重新排队 |
| `exhausted` | 已完成两遍有界修复，仍无可验证进展 |

### 5.3 两遍修复策略

第一遍的目标是尽快覆盖所有容易修复的失败：

1. 每个任务最多进行 2 次真实修改；
2. 每次修改后立即运行定向场景；
3. 通过后运行受影响 suite 的回归确认；
4. 连续无进展或达到 2 次仍失败时，标为 `deferred_no_progress`；
5. 立即处理下一个任务，不得让难例阻塞队列。

第一遍完成后运行全量验证，重建失败与 regression 队列。第二遍：

1. 为每个 deferred/regressed 任务启动新的诊断上下文；
2. 允许重新分类根因或进行有证据的架构级修复；
3. 每个任务最多再进行 2 次真实修改；
4. 第二遍结束仍无进展则标为 `exhausted`。

因此限制是“每个任务两遍、每遍最多两次”，不是“整个工程只能修四次”。动态发现多少失败任务，就分别给予相应的处理机会。

这 2+2 是单任务调度配额，不是阶段终止预算。第一遍用完配额只表示当前任务让出执行权；只有该任务完成第二遍且满足无进展证据时才是 `exhausted`。其他 queued task 不受它影响。

### 5.4 进展判定

一次修改出现任一情况可视为进展：

- 当前 task 中新增 passed scenario；
- 当前 task 的失败数量减少；
- timeout/crash/build/link 等失败推进到更深的 assertion 层；
- 失败指纹发生可解释变化，并且没有减少已确认通过场景；
- suite 总 passed 数增加。

以下情况属于无进展：

- 失败指纹完全相同；
- 只改变日志、状态字段、mapping 自述或注释；
- 定向场景未改善且 suite passed 数未增加；
- 新增通过与新增回归相抵后，评分向量没有提升；
- 当前文件没有真实变化。

### 5.5 验证粒度

为减少重复全量执行：

- 修改只影响单一场景时，先运行定向验证；
- 修改局限于一个动态 suite 时，定向通过后运行该 suite 回归；
- 修改公共模块、ABI facade、Cargo 配置或跨 suite 存储逻辑时，运行全量验证；
- 第一遍完成、第二遍完成、正常最终报告前必须运行全量验证；
- targeted 结果只能形成 provisional candidate，不能直接提升正式 submission-best。

### 5.6 测试侧必须按场景拆分门禁

逐用例优化要求 placeholder、Cargo execution、semantic review 和 consistency 都提供 per-scenario 结果：

1. placeholder 把缺失 test、无 assertion、空 mapping 等问题记录到对应 scenario task；
2. 已具备真实 test 体的场景允许执行定向 Cargo test，其他未就绪场景继续保留为 queued/fail；
3. semantic reviewer 允许接收一个 assigned scenario/最小 helper cluster，并把结果合并到按场景保存的 review ledger；
4. 每个 review case 自己的 fingerprint 必须覆盖它使用的 C test/helper、Rust test/helper 和 mapping item；
5. 一个 review case 失败不能让其他 current 且 pass 的 case 变成 unresolved；
6. consistency 的顶层 `status` 仍反映“是否全部通过”，但 `passed_scenarios` 必须按各 scenario 自身证据计算；
7. 每遍结束和正常 final 前，必须做全动态场景完整性检查，确保没有场景因为局部审查而被遗漏。

Rust 测试最终可计分 pass 至少要求同一 scenario 同时满足：

```text
真实 Rust test 存在且非占位
+ Cargo 实际执行并通过
+ 当前 semantic review case 通过
+ consistency case 无缺失/错配问题
```

### 5.7 IMPLEMENT 不能继续使用全局一次补漏

虽然逐用例队列主要服务 C-Cross 和 Rust tests，IMPLEMENT 的全局一次 repair 也会在进入用例阶段前造成提前终止，因此同步改为 finding task queue：

- 以 audit 动态 symbol/module/dependency finding cluster 建 task；
- 每个 task 只处理指定模块/符号并立即重审对应 finding；
- 一个 finding 难以解决时让出执行权，继续处理其他 scaffold residue；
- 所有 implementation finding resolved 后才能进入 C-Cross；
- 所有 finding 都完成有界两遍处理仍有 residue 时，才允许 IMPLEMENT `failed_final`；
- 540 分钟冻结仍优先恢复 submission-best/baseline，不提交当前半实现状态。

## 6. 两级 best-known 机制

### 6.1 `runtime-best`

保留现有 `logs/trace/c-cross/snapshots/` 机制，职责仅限于 C-Cross 修复期间保护生产 Rust 源码：

- 跟踪 `flashDB_rust/src/**`；
- 按 C-Cross 动态 passed 数和失败深度比较；
- 发生 regression 时恢复；
- 不作为第 540 分钟最终提交快照。

开发时应解除 `--restore-best` 只能紧跟 regression 的内部耦合，将底层“验证并恢复指定 best”提取为可复用函数；正常 C-Cross CLI 仍可保留 regression 限制。

### 6.2 `submission-best`

新增完整提交快照，目录为：

```text
logs/trace/submission-snapshots/
├── best.json
├── history.jsonl
└── <snapshot-id>/
    ├── manifest.json
    ├── score.json
    ├── files/
    └── evidence/
```

快照为不可变目录。创建成功后只通过原子替换 `best.json` 更新指针；不得原地覆盖旧快照。

### 6.3 文件范围

动态存在时纳入完整快照：

```text
flashDB_rust/Cargo.toml
flashDB_rust/Cargo.lock
flashDB_rust/build.rs
flashDB_rust/src/**
flashDB_rust/tests/**
logs/trace/rust_test_mapping.json
```

证据目录保存与评分相关、且指纹对应这些文件的只读副本，例如：

```text
implementation-audit.json
c-cross validation matrix/checkpoint
cargo-results.json
test-placeholder-check.json
test-semantic-review.json
test-consistency.json
unsafe-ratio.json（若已生成）
```

`target/**`、runner 临时目录、历史大日志、平台 C 输入和最终报告不得作为提交文件复制。最终报告在恢复后重新生成。

### 6.4 快照 manifest

`manifest.json` 至少包含：

```json
{
  "schema_version": 1,
  "snapshot_id": "submission-...",
  "run_id": "...",
  "created_at": "...",
  "scope": "submission",
  "files": [
    {"path": "flashDB_rust/src/...", "sha256": "...", "mode": 420}
  ],
  "evidence_fingerprint": {},
  "score_vector": [],
  "validation_scope": "full"
}
```

路径必须为工作台根目录下的规范相对路径；拒绝绝对路径、`..`、symlink 和超出白名单的文件。

### 6.5 快照候选等级

快照分为：

- `baseline`：当前 Rust 项目真实 build 成功，保证至少可恢复到可构建状态；
- `provisional`：定向场景改善，但未完成要求的 suite/full 回归；
- `validated`：完整评分输入与验证证据均为 current，可竞争 `submission-best`。

第 540 分钟只允许恢复 `validated`；若运行尚未产生 validated，则恢复最近的 buildable `baseline`。不得把 provisional 半成品作为最终提交。

## 7. 最优提交评分

### 7.1 候选资格

候选提升为 validated 前必须满足：

1. `cargo build` 对当前 Cargo 与 Rust 文件真实成功；
2. mapping、tests、Cargo 结果和 consistency 输入指纹对应当前文件；
3. mapping 非空时，映射测试均真实存在并被 Cargo 执行，不能以 0 tests 通过；
4. 相关验证范围完整，不得把未运行场景计为 passed；
5. 所有快照文件和证据文件 hash 已写入 manifest；
6. 快照复制完成并验证后，才可更新 `best.json`。

### 7.2 动态评分向量

不写死标准用例数量。根据当前 C model、mapping、Cargo execution 和 consistency 结果动态计算：

```text
score_vector = [
    build_success,
    judge_case_passed,
    consistency_passed,
    cargo_executed_passed,
    c_cross_passed,
    -not_run,
    -failed
]
```

按字典序比较：

1. build 成功的候选始终优先，避免恢复一个评分为 0 的不可构建状态；
2. `judge_case_passed` 是一致性门禁与真实 Cargo 结果共同判定的最终可计分用例数；
3. 一致性通过数优先于单纯 C-Cross 通过数；
4. C-Cross 用于在最终可计分数相同的情况下选择实现行为更完整的状态；
5. score 完全相同时，选择 `created_at` 更新的 validated 快照。

任何 stale、缺失、解析失败或范围不完整的证据都不能贡献 passed，只能计入 not-run/fail。

### 7.3 提升时机

以下时机尝试创建或提升 submission snapshot：

1. 首次获得 buildable Rust 工程时保存 baseline；
2. C-Cross 完成全量 checkpoint/confirmation 后；
3. Rust tests、placeholder、Cargo 和 consistency 完成 current 检查后；
4. 第一遍与第二遍全量回归后；
5. 正常 final 前。

只有评分向量提高，或评分相同但当前 validated 快照更新，才原子更新 `best.json`。

## 8. 无 Git 快照与恢复算法

### 8.1 创建快照

创建过程：

1. 解析动态白名单文件；
2. 拒绝 symlink、越界路径和缺失必需文件；
3. 复制到同一项目文件系统中的临时 snapshot 目录；
4. 对每个复制文件计算 SHA-256；
5. 写入并 fsync manifest、score 与 evidence；
6. 将临时目录原子 rename 为最终不可变目录；
7. 再次读取并验证 manifest；
8. 通过临时文件加 `os.replace` 原子更新 `best.json`；
9. 追加 `history.jsonl`。

快照失败不得破坏原 best 指针，也不得删除旧快照。

### 8.2 恢复快照

恢复过程：

1. 读取 `best.json`；
2. 验证 best 指向的 snapshot 存在；
3. 对全部缓存文件重新计算 hash；
4. 若 best 损坏，按 history 从新到旧寻找下一个相同最高分的有效快照；
5. 把文件复制到目标旁的临时文件；
6. 使用 `os.replace` 逐文件原子替换；
7. 删除 tracked scope 中 manifest 不包含的多余文件；
8. 再次计算当前提交文件 hash；
9. 写入 `logs/trace/submission-restore.json`。

恢复前必须先完整验证快照，不能边验证边覆盖当前工程。

### 8.3 恢复竞态

第 540 分钟可能恰好有 Agent 正在写文件。处理顺序固定为：

1. 先原子写 `FREEZE_REQUESTED`；
2. 所有受控工具停止接受新的修改；
3. 等待一个很短的文件写入收敛窗口，不等待业务任务完成；
4. 恢复完整快照；
5. 将提交范围切换为只读；
6. 连续两次检查当前 hash；
7. 若发现恢复后仍有漂移，再执行一次恢复并记录竞态。

只读范围包括 Rust 源码、测试、Cargo 配置和 mapping；`target/**`、`logs/trace/**` 的非 mapping 产物与 `result/**` 保持可写，以便生成报告。

## 9. 600 分钟看门狗

### 9.1 定位

看门狗不是 workflow controller，不负责任务调度、模型选择、验证结果解释或 repair 决策。它只有四个职责：

1. 记录单次运行的统一起点；
2. 在第 540 分钟发出冻结；
3. 恢复并冻结 submission-best；
4. 为 primary 提供“立即结束”的确定性状态。

### 9.2 启动

`INSTRUCTION.md` 的第一条命令必须是：

```bash
python3 work/tools/contest_watchdog.py start \
  --root . \
  --freeze-after-minutes 540
```

`start` 使用 `subprocess.Popen(..., start_new_session=True)` 启动后台 watch 进程，不使用 `/tmp`、shell `&`、`nohup` 或平台特有服务。父命令只有在收到 watchdog ready handshake 后才返回 0。

状态位于项目拥有目录：

```text
logs/trace/contest-control/
├── run.json
├── watchdog.log
├── ready.json
├── freeze-requested.json
├── submission-frozen.json
└── stop-requested.json
```

### 9.3 计时源

强制计时使用：

1. Linux `CLOCK_BOOTTIME`，包含系统 suspend 时间；
2. 若运行环境不支持，再退回 `time.monotonic_ns()`；
3. UTC wall-clock 只用于人类审计，不参与截止判断。

`run.json` 保存 `run_id`、boot id、watchdog PID、进程启动标识、单调时钟起点和审计时间。stage、primary 和 subagent 均不能重置该起点。

由于平台官方计时会略早于第一条工程命令，第 540 分钟冻结实际可能比平台第 540 分钟略晚；预留的 60 分钟用于覆盖该偏差和结束流程。若平台以后提供官方 deadline，应优先使用官方值。

### 9.4 单实例与陈旧进程

`start` 必须：

- 使用原子 lock 防止同一 run 启动两个 watchdog；
- 通过 PID、`/proc/<pid>/stat` 启动标识和 boot id 判断旧 PID 是否复用；
- 活跃 watchdog 存在时返回当前 run 状态，不重置计时；
- 旧 watchdog 已死亡时记录 stale recovery，再创建新 run；
- 正常完成后由 `stop` 写入停止信号并等待 watchdog 退出。

新一轮竞赛运行不得悄悄沿用上一轮的 elapsed，也不得删除历史 submission snapshots。

### 9.5 工具 guard

新增轻量共享 guard，供会修改工程或启动长任务的工具调用：

```text
guard_mutation_allowed(root)
guard_dispatch_allowed(root)
contest_state(root)
```

冻结后：

- scaffold、repair、test migration、mapping 修改、Cargo 修复和 subagent dispatch 均拒绝；
- 只读检查、快照恢复、报告生成和 FINAL gate 允许；
- 工具输出稳定标识 `CONTEST_FINALIZE_REQUIRED`，避免 primary 把拒绝解释成普通可修错误。

### 9.6 540 分钟冻结动作

到点后 watchdog 不询问 Agent，固定执行：

```text
write FREEZE_REQUESTED
        ↓
stop accepting mutations
        ↓
restore highest-score, newest-on-tie validated snapshot
        ↓
verify current file hashes and evidence fingerprint
        ↓
write deadline final evidence
        ↓
generate result/output.md and issue summary
        ↓
protect submission files from further writes
        ↓
write SUBMISSION_FROZEN
```

watchdog 不重新运行耗时的完整 C-Cross/Cargo/semantic repair loop。恢复的是此前已验证且由 hash 绑定的快照；评分平台会对最终文件独立测试。内部只进行快照完整性、当前文件 hash 和证据 fingerprint 检查。

### 9.7 primary 的结束义务

primary 在以下位置必须读取 contest state：

- 每次 subagent dispatch 前；
- 每次 subagent 返回后；
- 每次准备修改文件前；
- 每次验证工具返回后；
- 最终报告前。

看到 `FREEZE_REQUESTED` 或 `SUBMISSION_FROZEN` 后：

1. 不再分析新失败；
2. 不再恢复当前半成品；
3. 不再启动 subagent 或验证循环；
4. 若 watchdog 已生成报告，只运行允许的 FINAL 快速检查；
5. 立即结束任务并提交。

看门狗不杀死 OpenCode。通过 60 分钟余量、工具 guard、提交文件只读和“每个 subagent 只完成一次有界修改后返回”的合同，使 primary 有机会正常结束。若一个模型调用本身连续阻塞超过剩余 60 分钟，仅靠参赛工程无法主动生成模型的最终回复；因此所有单个任务必须保持短期、有界，不能把整个 suite 的连续调试交给一个 session。

## 10. deadline 终态与报告语义

### 10.1 正常终态

- 全部动态用例通过且 final evidence current：`success`；
- 所有失败任务均完成两遍处理、剩余任务为 exhausted，且完成正常全量确认：`failed_final`；
- 队列仍有未处理或可处理任务：`repair_required`，不得正常提前报告。

### 10.2 540 分钟终态

冻结点是竞赛明确的最终提交预算边界。到点后，恢复的 submission-best 形成受约束的 deadline 终态：

- 快照内全部用例通过：`success`；
- 快照仍有真实失败：`failed_final`，并写入 `termination_reason: contest_deadline_freeze`；
- 未通过、未运行和当时尚未处理的任务都必须如实列出，不能改写为 pass。

这里允许从运行中的 `repair_required` 进入 deadline `failed_final`，前提不是“Agent 决定不修”，而是同时满足：

1. 独立 watchdog 已证明到达 540 分钟；
2. 已恢复一个此前 validated 的完整提交快照；
3. 当前文件 hash 与该快照 manifest 完全一致；
4. 报告引用的是该快照的 current evidence bundle；
5. 提交已经冻结，后续不再修改。

缺少任一条件都不能伪造 deadline `failed_final`。

### 10.3 FINAL gate

`gate.py --stage FINAL` 增加两种合法证据路径：

```text
normal_final
deadline_snapshot_final
```

deadline 路径不要求恢复后再次进行完整 fresh runner 执行，但必须验证 watchdog、snapshot manifest、当前提交 hash、score/evidence fingerprint、restore evidence 和 frozen 状态形成闭环。

## 13. 异常与恢复

### 13.1 best pointer 损坏

- 不覆盖当前提交；
- 根据 history 从新到旧验证快照；
- 选择最高分且 hash 完整的候选；
- 记录 fallback 原因。

### 13.2 快照创建中断

- 临时目录不能写入 history 或 best；
- 下次启动清理仅属于当前 run 的未完成临时目录；
- 已完成快照保持不可变。

### 13.3 watchdog 重复启动

- 活跃 run 返回已有状态；
- stale PID 结合 boot id 和进程启动标识判定，不能只调用 `kill(pid, 0)`；
- 新 run 不复用旧 elapsed。

### 13.4 watchdog 到点但没有 validated 快照

- 优先恢复最近的 buildable baseline；
- baseline 也不存在时不得删除当前文件；
- 冻结当前状态，写明 `no_validated_snapshot`，尽快生成真实失败报告；
- 该情况必须作为开发缺陷进入 issue summary，因为正常流程应在首次 build 成功后立即建立 baseline。

### 13.5 恢复后 build 不一致

快照建立时已绑定 Cargo 和 Rust 文件。若恢复 hash 正确但评分平台重新 build 失败，报告保留真实失败；不得在冻结后修改快照。后续只能在下一次竞赛运行修复 snapshot eligibility 或环境兼容问题。

## 14. 安全边界

1. 所有缓存都位于工程自己的 `logs/trace/`，不得使用 `/tmp/**` 或 `/var/tmp/**`。
2. 不读取、复制或修改平台 C 输入到 submission snapshot。
3. 拒绝 symlink 和路径穿越。
4. restore 只处理白名单文件；不得删除整个项目根目录。
5. 不执行 `git reset`、`git checkout`、`git stash` 或假设存在 `.git`。
6. 不把日志、自述状态或 mapping 字段当成 passed 证明。
7. deadline final 必须保留真实失败，不能通过删除 tests、减少 mapping 或跳过场景提高表面分数。
8. watchdog 不发送网络请求，不依赖外部服务。

## 15. 回归测试与验收标准

### 15.1 工作队列

1. 初始失败动态生成 queued tasks，不硬编码数量。
2. 一个任务第一遍连续两次无进展后被 deferred，后续任务仍会执行。
3. 第一遍结束后 deferred 任务进入第二遍，而不是直接 `failed_final`。
4. 第二遍耗尽后仍失败才能标为 exhausted。
5. 任一未处理 task 存在时，正常 gate 仍为 `repair_required`。
6. 一个 cluster 修好多个场景时，相关场景分别变为 provisional/confirmed pass。
7. 新修改造成已确认场景回归时，任务重新排队并恢复 runtime-best。

### 15.2 submission snapshot

1. 不存在 Git 仓库时仍能 capture、比较、restore。
2. `src/**`、`tests/**`、Cargo 和 mapping 能完整恢复。
3. 恢复会删除 tracked scope 中的额外文件。
4. 修改缓存文件后 hash 校验失败，不使用损坏快照。
5. best 损坏时能从 history 回退到有效快照。
6. build 失败候选不能替换 build 成功 best。
7. 空 tests、0 tests 或 stale consistency 不能提高 score。
8. 一致性通过数增加时能提升 best；完全同分时选择更新 validated 快照。
9. provisional 快照不能成为 deadline restore 目标。
10. snapshot 创建中断不破坏原 best。

### 15.3 watchdog

1. 使用可注入的短测试时长验证冻结，不在单测中等待 540 分钟。
2. 同一 run 重复 start 不重置单调起点。
3. stale PID、PID 复用和 boot id 变化能被识别。
4. wall clock 人为前后跳变不影响冻结时机。
5. 到点先写 freeze，再恢复 best。
6. 冻结后 mutation/dispatch 工具返回 `CONTEST_FINALIZE_REQUIRED`。
7. 冻结后提交文件 hash 连续检查一致。
8. watchdog 正常 stop 后不再触发冻结。
9. watchdog 不 kill OpenCode，也不修改平台输入。

### 15.4 deadline final

1. 到点时当前工作区即使是更低分半成品，也恢复最高分 validated snapshot。
2. 最高分相同的多个快照恢复最新一个。
3. deadline final 的失败场景仍写为 failed/not-run。
4. 没有 freeze evidence 时，不能使用 deadline final 绕过 `repair_required`。
5. 恢复文件 hash 与 snapshot manifest 不匹配时 FINAL gate 失败。
6. 正常 success/failed_final 路径不受 watchdog deadline 路径影响。
7. `result/output.md`、issue summary 和 deadline evidence 在冻结后能够生成。

### 15.5 全量回归

实施完成后至少运行：

```bash
python3 -m unittest discover -s 02_02/tests -p 'test_*.py'
```

并增加独立短时集成测试，在临时工作台副本中验证：

```text
创建 baseline
→ 创建更高分 validated snapshot
→ 制造低分/不可构建半成品
→ 触发短时 watchdog freeze
→ 恢复最高分版本
→ 验证只读冻结、报告与 FINAL gate
```

## 16. 实施顺序

按以下顺序开发，避免先修改 Agent 合同却没有工具支撑：

9. 修改 gate/report_writer 的正常终态与 deadline snapshot 终态。
10. 最后精简更新 `INSTRUCTION.md`、orchestrator 和两个修改型 subagent 文档。
11. 运行单元测试、短时 watchdog 集成测试和一次不依赖 Git 的端到端演练。

## 17. 开发完成判定

同时满足以下条件才可把本文档状态改为“已实施”：

1. 不依赖 Git 的完整 submission snapshot 可创建、比较、恢复和校验。
2. 正常修复按动态任务队列推进，一个难例不会阻塞后续任务。
3. C-Cross runtime-best 与最终 submission-best 职责分离。
4. 动态评分向量与 build、用例一致性和真实 Cargo 执行结果一致。
5. watchdog 从首条命令启动，单一 run 不会重置计时。
6. 短时测试证明到点能够恢复最高分且同分最新的 validated 快照。
7. 冻结后无法继续修改提交文件，primary 会进入立即结束路径。
8. deadline final 不能绕过 hash、evidence 和 freeze 证据。
9. 正常 success、正常 failed_final 和 deadline failed_final 都能生成真实报告。
10. 全量回归测试通过，`.opencode` 镜像已同步。
