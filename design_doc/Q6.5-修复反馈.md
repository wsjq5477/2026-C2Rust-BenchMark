## 当前结果的正确口径

- 原始矩阵：`validation-matrix.json` 记为 10 项通过、14 项失败；其中 11 个 TSDB 条目是 `c_cross_result_parse_failed`，不是 11 个已证实的业务失败。
- 隔离复跑：以行缓冲在 `/tmp` 重跑同一 `tsdb_runner` 后，确认 TSDB 前 3 项通过，两个 iterator 用例失败，`query_count` 通过，`set_status` 场景超时，后 4 项未执行。

本轮可确认的完整结果为：14 项通过、5 项明确断言失败、1 项超时、4 项未运行。

| 用例簇 | 当前结果 | 归因结论 |
|---|---:|---|
| KVDB 基础 init/blob/增删改/default/deinit | 10 通过 | ABI、基础存储和 blob 生命周期已可用 |
| KVDB GC / 扩容 | 3 失败 | 多扇区 live record 恢复、`oldest_addr` 重建或控制命令到存储几何同步存在缺口 |
| TSDB init/clean/append/query_count | 4 通过 | 基础 TSDB 存储、查询范围统计可工作 |
| TSDB iterator / iterator_by_time | 2 失败 | callback 已被调用；失败点是回调对象 `tsl->time` 与负载期望时间不一致，不是函数指针类型映射失败 |
| TSDB set_status | 1 超时 | 卡在该场景内部，尚无单 API 级证据，需细化运行诊断 |
| TSDB 后续 clean/跨扇区迭代/deinit/issue_249 | 4 未运行 | 被 set_status 超时阻断，当前不能判定通过或失败 |

KVDB 三个失败用例共享“多 KV/大 blob 压力、重启后恢复、地址对齐、GC 或存储扩容”语义；因此根因不是基础读写，而是跨扇区恢复与 live/tombstone 选择链。现有 trace 尚不能在“扫描遗漏、最新版本选择错误、删除状态恢复错误、`oldest_addr` 重算错误、控制命令仅更新 facade”之间进一步定责。

TSDB 的 `query_count from=0 to=512 => 256, expected=256` 已通过，而两个 iterator 的 `tsl->time == atoi(data)` 失败；这表明时间范围统计并未整体失效，优先怀疑 iterator 输出对象的时间字段重建、记录读取路径或迭代顺序。`set_status` 超时是独立的状态转换/迭代交互问题，不能与两个时间断言混为同一根因。

## 问题项

### Q6.5-1：TSDB 超时导致 C-cross 结果被错误放大

#### 问题描述

- 现象：TSDB runner 超时后，stdout 中缺少 `Running:` 行；解析器将整个 suite 的 11 个 scenario 标为 `c_cross_result_parse_failed`。
- 影响：真实通过、真实失败和未执行用例混为一谈，repair 无法准确定位；阶段记录中“TSDB 7 PASS”也没有原始 runner 证据支撑。
- 证据：原 trace 的 `tsdb_runner.log` 只记录两个 FAIL 和 timeout；隔离行缓冲复跑则记录到 `init_ex`、`clean`、`append`、`query_count` 已完成，且在 `set_status` 开始后超时。
- 根因判断：已证实为 runner 超时场景下的输出缓冲与 parser 严格依赖 `Running:` 行共同造成；两个迭代时间断言失败和 set-status 超时是独立的业务现象，后续 4 项状态未知。

#### 解决思路与预期效果

- 解决思路：以行缓冲或 PTY 执行 runner；超时时保留“已开始、已失败、未开始”三类结果，不能把未知场景伪装为测试失败。
- 预期效果：下一轮矩阵能精确给出已知失败、未执行和真实通过；TSDB 修复可优先处理时间迭代断言及实际挂起点。
- 不采用的方案：不通过修改 C runner 增加日志，避免触碰比赛输入。

#### 设计方案

1. 修改范围：`work/tools/c_cross_validate.py` 及其框架测试。
2. 数据/产物合同：新增或细分 `failure_layer`/`diagnosis`，至少区分 `assertion_failed`、`runner_timeout`、`result_protocol_error`；未开始场景使用 `not_run`。每个失败或超时条目附带来自 `c_test_model` 的 obligations、数据规模、调用 API、注册顺序和原始输出片段。
3. 执行流程：优先以 `stdbuf -oL -eL` 运行；不可用时使用 PTY 或记录无行缓冲降级模式；parser 在异常退出时仍保存已观察到的 PASS/FAIL，并把未开始场景与已开始未完成场景分开。
4. Gate 与验收：构造“两个 FAIL 后 timeout 且无 Running 行”的单元测试；新矩阵必须得到 14 pass、5 assertion fail、1 timeout、4 not_run 的等价分类；final gate 继续拒绝 `fail`、`timeout` 与 `not_run`，中间 gate 接受真实诊断。
5. 风险与兼容性：保留现有正常 runner 的解析规则，capture mode 写入日志，避免平台缺少 `stdbuf` 时静默降级。

### Q6.5-4：ABI 仅覆盖顶层结构体，未形成嵌套类型闭包

#### 问题描述

- 现象：`c_api_model.json.abi_layouts` 包含 9 个公开顶层结构体，但缺少 `kv_cache_node`、`kvdb_sec_info`、`tsdb_sec_info`。
- 影响：顶层 layout probe 可以通过，但嵌套字段的独立结构体布局没有模型证据，后续实现或配置变化仍可能错误。
- 证据：当前 ABI layout 列表不含上述 3 个嵌套结构体；`fdb_kvdb`、`fdb_tsdb` 的 layout probe 已通过。
- 根因判断：已证实为 extractor 仅以公开 `fdb_*` 结构体作为候选，未递归展开字段依赖。

#### 解决思路与预期效果

- 解决思路：从公开 ABI 结构体字段、数组元素和 typedef 指向类型递归收集结构体依赖，形成 layout 闭包。
- 预期效果：Rust FFI 类型设计拥有每个实际嵌套结构体的编译器确认尺寸、对齐和字段偏移。

#### 设计方案

1. 修改范围：`work/tools/abi_layout_extractor.py`、`work/tools/build_c_model.py`、`work/tools/gate.py`。
2. 数据/产物合同：`abi_layouts` 增加 `dependency_of` 或等价来源字段；模型记录完整 ABI 闭包。
3. 执行流程：预处理后解析字段类型并递归加入候选，再统一编译 probe。
4. Gate 与验收：公开结构体引用的非基础结构体必须存在 layout；对本输入至少覆盖 `kv_cache_node`、`kvdb_sec_info`、`tsdb_sec_info`。
5. 风险与兼容性：允许无定义的前置声明保持指针类型，不要求为不完整类型生成 `sizeof`。

- 解决思路：明确采用“C 可见 facade 字段合同 + Rust sidecar 状态”策略，并要求 rewrite 批次完整覆盖设计要求。
- 预期效果：状态所有权、生命周期和同步边界明确；所有动态语义要求都有机器可读实施证据。
- 不采用的方案：不采用把 Rust 对象嵌入 C 结构体的方案；当前 ABI 没有安全保留空间，`user_data` 属于调用方。

3. 执行流程：init/deinit/重复 init/指针复用/回调期间锁行为均由设计明确；REWRITE 根据 requirement ID 分批记录。
4. Gate 与验收：设计策略必须合法；批次 obligations 并集必须覆盖全部 requirement ID；控制、持久化、GC、blob、iterator、status 均有对应 C-cross 证据。

### Q6.5-8：测试语义没有沿 iterator callback 和辅助函数传播

#### 问题描述

- 现象：`test_fdb_tsl_iter` 和 `test_fdb_tsl_iter_by_time` 在 `c_test_model.json` 中的直接断言数为 0，但真实 runner 在 callback 中断言 `tsl->time == atoi(data)`；`test_fdb_tsl_set_status` 的 public API 列表也未出现状态设置 API。
- 影响：设计阶段只能生成泛化的 iterator/status 义务，无法要求“回调对象时间字段与负载一致”，也无法发现 callback 内变更状态可能产生的锁重入或无终止迭代风险。
- 证据：隔离行缓冲复跑在两个 iterator 用例中输出时间断言；相应 scorer case 的 `semantic_facts.assertion_details` 为空。set-status 场景的 `semantic_facts.public_api_calls` 仅含 iterator/query 路径，随后该场景超时。
- 根因判断：已证实为 `build_c_model.py` 以测试函数直接调用为主，未递归展开传给 callback 的函数体、辅助函数调用链及其中的断言/API。

#### 解决思路与预期效果

- 解决思路：对 C 测试中的函数指针实参建立 callback call graph，把 callback 和其辅助函数中的断言、字段访问、数据库 API 调用回填到所属 scenario。
- 预期效果：实现阶段获得精确的 `iterator_time_payload_consistency`、`callback_object_field_fidelity` 与 `callback_reentrancy` 要求；C-cross 超时可以按实际 callback 链路路由修复。

#### 设计方案

1. 修改范围：`work/tools/build_c_model.py`、`work/tools/design_rust_api.py`、`work/skills/rust-implementer.md`、`work/tools/gate.py`。
2. 数据/产物合同：scenario 新增 `callback_paths`；每条记录包含 iterator API、callback 名称、传入上下文、callback 内 API、字段断言和辅助函数链。
3. 执行流程：BUILD_C_MODEL 解析函数指针实参并递归收集限定深度的 callback/helper body；DESIGN_RUST_API 将结果转为精确 implementation requirements；REWRITE 批次按 requirement ID 记录实现。
4. Gate 与验收：iterator scenario 必须包含 callback 证据；如果 callback 调用状态变更 API，设计必须声明回调期间锁策略或快照策略；本输入必须捕获 `tsl->time` 时间一致性断言和状态设置链路。
5. 风险与兼容性：递归需有去重与深度上限，遇到未知 callback 时记录 unresolved evidence，不能静默丢弃。
