# Q9 - FlashDB C-to-Rust 转换工程架构简化设计

> 状态：简化验证版已实施，待与复杂控制版进行 A/B 对照
> 日期：2026-07-14  
> 范围：`02_02` 转换工程的控制面、验证面与执行边界；不修改平台 C 输入，不预置 Rust 答案。

## 1. 结论

下一轮优化不再向当前工作流增加 gate、状态字段、task packet、worker 角色或 repair 脚本，而是以“保留真实验证、删除过程编排”为原则收缩工程。

目标不是回退到早期的弱验证版本，而是形成下面的最小闭环：

```text
一个主 Agent
    ↓
动态提取 C 事实与 ABI 事实
    ↓
生成骨架并实现 Rust
    ↓
真实 C 编译 / 链接 / 运行验证
    ↓
围绕失败场景修复并复跑
    ↓
迁移 Rust 测试、cargo 验证、如实报告
```

主 Agent 是默认执行者；subagent 只能是可选的、短生命周期的上下文隔离手段，不能成为阶段推进、产物落盘或最终通过的前置条件。

## 2. 当前问题

当前工程已从“迁移工作台”逐渐变成“事务控制平台”。除 C 建模、ABI 校验、C-Cross、测试迁移外，还维护了：

- `workflowctl` 的 stage run、nonce、receipt、hash、next action；
- task packet 及其输入、路径、大小和身份合同；
- REWRITE 的 parent run、CORE/FACADE worker、revision、receipt 和重检；
- subagent registry、invocation、fallback 记录；
- 多份 stage 文档、JSON 合同、gate 规则和测试断言的同步。

这些机制可以证明“某个预设过程发生过”，但不能直接提高 Rust 行为与原 C 测试的一致性。反而会带来三类新失败：

1. 模型需要理解更多命令、状态和回执，减少了用于实现与修复的上下文；
2. subagent 写入、回执和主工作区之间存在额外交接风险；
3. 一处流程缺陷通常会新增一层合同和回归测试，旧层不删除，复杂度单调累积。

Q3 已确认过正确方向是收紧读取边界，而不是新增 packet 或状态机；Q7 的实际对比也显示，决定结果的是实现架构和是否持续消费 C-Cross 失败，而不是过程证据是否更细。

## 3. 设计目标与非目标

### 3.1 目标

1. 所有成功结论都能回到真实外部证据：ABI probe、原 C runner、Rust build/test 和测试映射。
2. 一个运行只有一个默认执行者，跨阶段语义不因角色边界而断裂。
3. 失败反馈直接给实现者：失败场景、失败层、最小日志 tail、相关源文件，而不是新的任务协议。
4. C 文件、符号、测试数量和测试语义仍从当前输入动态提取；不得写死样例答案。
5. 新机制只能替换已有机制，不能只叠加。

### 3.2 非目标

- 不删除真实 C-Cross harness、ABI layout probe、输入不可变校验、测试一致性检查或最终报告。
- 不把 FlashDB 迁移退化为仅 `cargo check` 或文本扫描。
- 不要求主 Agent 一次会话完成所有工作；需要缩短上下文时，按“实现阶段 / 验证阶段”切换会话即可，不引入 worker 事务协议。
- 不把当前本地样例的文件名、函数名、case 数或最佳 Rust 实现写进通用合同。

## 4. 目标架构

```text
INSTRUCTION.md（短入口：目标、输入、输出、禁止事项）
    │
    ▼
primary migration skill（唯一默认执行者）
    │
    ├─ ANALYZE
    │    ├─ scan_c_project.py
    │    ├─ build_c_model.py
    │    ├─ abi_layout_extractor.py
    │    └─ design_rust_api.py
    │
    ├─ IMPLEMENT
    │    ├─ generate_rust_scaffold.py
    │    └─ 主 Agent 根据动态事实实现与修改 Rust
    │
    ├─ VERIFY_AND_REPAIR
    │    ├─ ABI layout check
    │    ├─ c_cross_validate.py（真实 C compile/link/run）
    │    └─ 失败场景 → 最小代码修复 → 同一工具复跑
    │
    └─ TEST_AND_REPORT
         ├─ migrate_tests.py
         ├─ placeholder / consistency 检查
         ├─ cargo_capture.py
         └─ report_writer.py
```

阶段是面向 Agent 的工作分组，不是事务数据库状态。阶段能否继续由当前产物和验证命令的退出结果判断；不以 receipt、nonce、调用记录或自然语言“已完成”为依据。

## 5. 证据与状态的边界

### 5.1 保留的机器证据

以下产物直接回答迁移是否真实，应保留并由工具生成：

| 事实 | 权威产物 |
|---|---|
| 当前 C 输入与不可变基线 | `input_manifest.json`、`input-source-baseline.json` |
| API、测试、ABI 事实 | `c_project_model.json`、`c_api_model.json`、`c_test_model.json` |
| Rust 设计与生成边界 | `rust_api_design.json`、`scaffold-manifest.json` |
| ABI 是否一致 | `layout-check.log`、layout check JSON |
| Rust 是否真能被原 C 测试调用 | `validation-matrix.json`、`cross-compile.log`、`cross-test.log` |
| Rust 测试是否逐项覆盖 | `rust_test_mapping.json`、placeholder / consistency JSON |
| 最终可构建与可测试 | cargo 日志和结果 JSON |
| 最终结论 | `result/output.md`、`result/issues/00-summary.md` |

### 5.2 删除的过程状态

下列信息不能作为通过依据，也不应成为运行必需产物：

- stage run、stage receipt、nonce、root inode、artifact hash 对账；
- task packet、packet hash、packet 限额和 packet 身份链；
- REWRITE parent/worker run、worker receipt、revision、recheck 状态；
- subagent registry、invocation、失败三次后的 fallback 资格；
- controller 根据旧状态推导的 `next_action`。

如需查看进度，主 Agent 直接检查上述真实产物并执行相应验证命令。进度是派生信息，不是第二套事实源。

## 6. 工具收缩方案

### 6.1 保留并收紧职责

| 能力 | 处理原则 |
|---|---|
| C 扫描、模型提取、API 设计 | 保留；只做动态事实提取与设计材料生成，不负责控制流。 |
| ABI layout 提取与 checker | 保留；它是编译器级证据，失败必须阻断功能验证。 |
| scaffold 生成 | 保留；只生成可编译结构、ABI 声明和可观察义务入口，不替模型完成业务实现。 |
| C-Cross | 保留为最强门控；只接受真实 staticlib + 原 C runner 的结果。 |
| 测试映射、一致性、占位检查 | 保留为最终测试真实性验证，不要求每个中间阶段写额外回执。 |
| cargo 与报告工具 | 保留；报告必须在成功或失败时都生成。 |

### 6.2 删除或吸收

| 当前机制 | 目标处理 |
|---|---|
| `workflowctl.py` | 删除。主控按短入口和真实验证命令推进，不维护事务控制器。 |
| `task_packet.py` | 删除。当前阶段的动态事实、失败摘要和允许编辑范围由主控在短提示中给出。 |
| `rewrite_check.py` | 合并到 IMPLEMENT 后的 cargo/ABI/C-Cross 验证，不保留独立 worker check 协议。 |
| `c_cross_converge.py`、`repair_planner.py` | 删除或并入 `c_cross_validate.py` 的简短失败摘要；工具只报告，不决定谁、何时、以何种角色修复。 |
| `test_failure_triage.py` | 只在其能从 cargo 结果产生不可替代分类时保留；否则由 `cargo_capture.py` 输出简短摘要。 |
| `gate.py` | 收缩为少量结果验证函数或直接由既有专用工具返回退出码；删除 stage receipt、agent、packet 和控制器状态校验。 |

不要求一次性拆分或新建更多脚本。优先删除编排代码，把仍需保留的结果检查放回已有专用工具。

## 7. Agent 与 subagent 规则

### 7.1 默认规则

主 Agent 负责分析、实现、验证、修复和报告。它不需要先证明调用了 subagent，才有资格修改 Rust 或推进流程。

### 7.2 可选隔离

仅在下列条件同时成立时才可使用 subagent：

1. 任务可在一个明确文件范围和明确停止条件内完成；
2. 主工作区可立即验证其写入结果；
3. subagent 不持有流程状态，也不写通过结论；
4. subagent 不成功时，主 Agent 可直接接手，不需要等待次数阈值。

适合的例子是“从当前 C 测试模型生成 Rust test 初稿”或“根据一组 C-Cross 失败场景修改指定 Rust 文件”。不适合把一个完整阶段拆成若干强制角色、状态链和回执链。

### 7.3 上下文控制

上下文问题只使用三项规则解决：

- 大模型 JSON、全量历史日志和总设计文档是冷数据；
- 从 `rg` 定位后读取局部 C/Rust 窗口；
- 需要新会话时，从当前文件、失败摘要和验证产物恢复，不传递旧对话历史。

不再用 packet 字段、worker revision 或控制器协议替代上下文管理。

## 8. 最小门控集

门控只保留与最终真实性有直接因果关系的四组。

1. **输入与模型门控**：C 输入未被修改；模型包含动态发现的测试、关键 ABI 和未解决项。
2. **ABI 门控**：Rust 导出的 layout 与 C 编译器 probe 一致；不通过时不启动功能 runner。
3. **行为门控**：原 C runner 已完成编译、链接和运行；矩阵逐场景记录通过、失败或解析失败，不能把 `not_supported` 当通过。
4. **最终门控**：Rust 测试映射、一致性、占位检查、cargo build/test、unsafe 比例和最终报告相互一致。

中间实现不因“已经写日志”“调用过 agent”“目录中存在 JSON”而通过。C-Cross 失败时保持失败，但仍允许进入最终报告以输出真实比赛产物。

## 9. 迁移步骤

本设计实施时必须从新分支开始，不能在当前复杂控制器上继续叠加兼容层。

### 阶段 A：冻结并建立对照

1. 保留当前提交 `4c00c9c` 作为复杂控制版基线。
2. 选取已有、尚未引入 `workflowctl/task_packet` 的工作台版本作为简化候选起点；这不是直接回退，后续 ABI、runner 隔离和真实性检查按能力迁移。
3. 固定同一份平台 C 输入、相同模型配置和相同运行预算，分别运行复杂控制版与简化候选版。

### 阶段 B：先删合同与测试负担

1. 将 `INSTRUCTION.md` 缩为目标、输入/输出、四阶段和禁止事项。
2. 将主控 skill 改为单主 Agent 流程；subagent 改为可选能力。
3. 删除测试中只断言 packet、receipt、agent 注册、nonce、worker revision 或旧路径的条目。
4. 保留并加强测试中验证真实 C-Cross、ABI、输入不可变、Rust 测试真实性和最终报告一致性的条目。

### 阶段 C：删除控制器，保留验证器

1. 删除 `workflowctl.py` 和 `task_packet.py` 的调用入口及镜像合同。
2. 删除其专有 trace 目录和 schema。
3. 把剩余的结果验证迁入专用工具或收缩后的 gate。
4. 删除 `REWRITE_CORE_MODULES` 内部 worker/revision 协议，恢复“实现 → C-Cross → 修复”的短循环。

### 阶段 D：端到端验收

每次运行只以最终真实产物判断，不以工作流日志是否漂亮判断。至少比较：

- C-Cross 动态场景的通过数与失败数；
- 是否能完成到最终报告；
- Rust 测试映射是否覆盖动态发现的 C 场景；
- ABI、cargo build/test 与报告是否一致；
- Agent 工具调用数、失败的流程命令数和总运行时间。

## 10. 变更准入规则

以后任何“优化”在合入前必须满足以下至少一项，同时不得新增无法删除的平行控制面：

- 直接提升真实 C-Cross 或最终测试的通过率；
- 直接降低一次完整运行的上下文、工具调用或耗时；
- 消除一个已发生的、可复现的真实性漏洞；
- 替换并删除至少一个现有脚本、状态文件、合同段落或流程测试。

如果一个建议只是“让 Agent 更难绕过流程”，却不能说明如何提高真实行为证据，默认不实施。

## 11. 审核点

实施前需要确认以下决策：

1. 是否接受“单主 Agent 为默认，subagent 可选”的执行模型。
2. 是否接受删除 `workflowctl.py`、`task_packet.py` 和 REWRITE worker/revision 协议，而不是继续修补它们。
3. 是否接受将回归测试的重点从内部流程合同转为端到端真实验证。
4. 是否接受先做复杂版与简化版的对照运行，再决定是否迁移全部现有有效能力。

在上述审核完成前，本设计不授权修改 `02_02/work`、`.opencode` 镜像、工作流测试或运行时合同。

## 12. 实施记录

2026-07-14 已在分支 `q9-simplification-validation` 实施第一版：删除事务 controller、task packet、REWRITE worker/revision、repair route 及其流程测试；入口改为单主 Agent 四阶段；C-Cross 输出只保留动态 `failure-summary.json`。

实施后 smoke 从动态 design 生成 scaffold，验证 Rust `staticlib` 构建通过、KVDB/TSDB 原 C runner 链接通过并实际执行。该 scaffold 尚未完成业务实现，C-Cross 结果为动态场景的部分通过与失败；这条结果用于证明真实执行链仍在，不构成迁移成功结论。
