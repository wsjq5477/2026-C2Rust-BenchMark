## 背景与范围

2026-07-14 的 OpenCode/GLM-5.1 运行中，`rust-implementer` 在 `REWRITE_CORE_MODULES` 阶段运行到 step 202 后失败。服务端返回：输入长度必须位于 `[1, 202745]`，实际请求超过上限。

| 指标 | 实际值 |
|---|---:|
| 模型 step | 202 |
| message | 204 |
| tool call | 226 |
| Bash | 151 |
| Read | 38 |
| Write/Edit | 36 |
| 累计 input tokens | 375,003 |
| 累计 output tokens | 72,966 |
| OpenCode compact 状态 | 未触发，`time_compacting=null` |

该 session 同时承担了内部 Rust 实现、外部接口接线、编译修复、format、audit、回执修复和 baseline 排障。单次 `cargo fmt --check` 输出约 96KB，build 输出最高约 61KB。问题不是某一个初始 prompt 过大，而是一个 session 承担了完整 REWRITE 事务及所有失败历史。

本方案只修改 C-to-Rust 转换工程的 scaffold manifest、workflow controller、task packet、check wrapper、gate、skill 和 trace 合同，不修改 C 输入，也不直接修改最终 Rust 产物。

## 设计结论

不再实现动态 requirement 分批、依赖图拓扑排序、语义聚类或多 agent 协作。外部仍保留一个 `REWRITE_CORE_MODULES` 阶段，内部固定为两类顺序 worker：

```text
REWRITE_CORE_MODULES
  │
  ├─ Controller PREPARE
  │    ├─ 保存 scaffold baseline hash
  │    ├─ 读取 scaffold manifest 的 rewrite ownership
  │    └─ 初始化静态 revision 状态
  │
  ├─ IMPLEMENT_CORE worker
  │    └─ 只修改 core-owned 内部实现
  │
  ├─ Controller CORE_CHECK
  │    ├─ 通过：进入 facade
  │    └─ 失败：重试 IMPLEMENT_CORE
  │
  ├─ WIRE_FACADE worker
  │    └─ 只修改 facade-owned 外部接口函数体
  │
  ├─ Controller FINAL_CHECK
  │    ├─ facade 问题：重试 WIRE_FACADE
  │    ├─ core 问题：重试 IMPLEMENT_CORE，使 facade 回执失效并重验
  │    └─ frozen/shared 问题：阻断，不在 REWRITE 修改
  │
  └─ Controller FINISH
       ├─ 对账真实 diff 和回执修订链
       ├─ 运行最终 audit/build/gate
       └─ 生成阶段证据
```

OpenCode/GLM 不负责划分文件、识别内外接口、选择下一个 worker 或修改流程。正常的 CORE→FACADE 前向交接不传递对话和自然语言实现总结；当前工作树、当前 packet 和 controller-owned receipt 是唯一交接介质。只有 facade 明确报告缺少内部能力时，允许在 controller-owned receipt 中携带不超过 1000 bytes 的单条 bounded reason 回到 CORE，不携带原 session 历史。

## 四类所有权合同

四类归属由 `generate_rust_scaffold.py` 在生成文件时写入 `scaffold-manifest.json`。controller 只消费和校验 manifest，不根据项目名、模块名、目录名或业务关键词猜测归属。

| 类型 | 写入者 | 来源 | 处理规则 |
|---|---|---|---|
| `core_owned_paths` | `IMPLEMENT_CORE` | scaffold 生成的内部 placeholder 和内部支持文件 | facade 可读不可写 |
| `facade_owned_paths` | `WIRE_FACADE` | scaffold 生成的外部函数 stub 所在文件 | core 可读不可写 |
| `frozen_contract_paths` | 无 | ABI 类型、layout probe、Cargo/生成配置等已确认合同 | REWRITE 中任何 worker 都不可修改 |
| `shared_readonly_paths` | 无 | 模块入口、兼容 re-export 和共享声明 | 两类 worker 都可读不可写 |

建议 manifest 合同：

```json
{
  "rewrite_ownership": {
    "schema_version": 1,
    "core_owned_paths": ["{generated-path}"],
    "facade_owned_paths": ["{generated-path}"],
    "frozen_contract_paths": ["{generated-path}"],
    "shared_readonly_paths": ["{generated-path}"],
    "frozen_contract_symbols": [
      {
        "symbol": "{generated-symbol}",
        "path": "{generated-path}",
        "signature_sha256": "{generated-hash}"
      }
    ]
  }
}
```

### 为什么需要 `frozen_contract_symbols`

仅用文件级四分类不够。facade 文件同时包含：

- 必须修改的函数体；
- 不得修改的函数名、ABI、参数和返回类型。

因此 facade 文件归 `facade_owned_paths`，同时由 `frozen_contract_symbols` 保护其签名。Gate 必须从当前源码重新计算签名 hash，不信任 agent 自报。

### 四类合同可行性结论

当前 scaffold manifest 已有两类可复用证据：

- `placeholder_modules[*].path` 可作为 `core_owned_paths` 的基础；
- `ffi_functions[*].path` 和 `stub_symbols` 可作为 `facade_owned_paths` 及冻结符号的基础。

但当前 manifest 还没有完整四分类，`workflowctl` 也没有静态 worker 子事务。所以方案在工程上可实现，但尚未实现，不能只通过修改 prompt 生效。

## 修复与“回滚”原则

当前 `workflowctl` baseline 记录的是路径和内容 hash，不保存可恢复的源码副本。`workflowctl abort` 恢复的是 workflow state，不是 worker 已修改的 Rust 文件。因此本方案明确不设计自动源码回滚，也禁止使用 Git、`/tmp`、`cp` 备份或重新生成 scaffold 来回退。

采用“原地向前修复 + 下游回执失效重验”：

```text
core revision 1 通过
  → facade revision 1 失败并报告 missing_core_capability
  → 重试 IMPLEMENT_CORE，生成 core revision 2
  → facade revision 1 回执因 based_on_core_revision=1 自动失效
  → controller 重新检查 facade
  → 若已通过，生成新验证回执；若失败，重试 WIRE_FACADE
```

任何 revision 都不删除旧证据，而是追加新回执并用 `supersedes` / `based_on_*_revision` 建立修订链。最终 gate 只接受基于最新 core/facade revision 的检查证据。

## 问题项

### Q8-1：REWRITE 是单 session 超大事务

#### 问题描述

- 现象：一个 subagent 同时实现内部逻辑、外部接线、编译修复、audit 和工作流排障，202 step 后超过上下文上限。
- 影响：已写入的实现无法稳定完成阶段回执，弱模型完成率明显下降。
- 根因：实现、接线、验证和控制器职责没有隔离。

#### 解决思路与预期效果

- 解决思路：固定拆为 `IMPLEMENT_CORE` 和 `WIRE_FACADE` 两类顺序 worker，其他工作交给 controller。
- 预期效果：每个 session 只理解一类代码责任，不继承前一 session 的对话、命令输出和排障历史。

#### 设计方案

1. `IMPLEMENT_CORE` 只获得 `core_owned_paths` 写权，负责内部状态、行为和 Rust API。
2. `WIRE_FACADE` 只获得 `facade_owned_paths` 写权，负责参数/返回值转换、空指针检查、panic 边界和调用内部 API。
3. 两类 worker 都不生成 receipt、不写阶段报告、不处理 gate/controller 问题。
4. 没有第三类 `INTEGRATION_REPAIR` agent；错误必须返回原所有者修复。
5. 验收：trace 中一个 worker session 不得同时拥有 core 和 facade 写权。

### Q8-2：内外部边界不能由弱模型判断

#### 问题描述

- 现象：如果只在 prompt 中要求 agent 判断内部模块与外部接口，不同模型可能产生不同分类。
- 影响：两个 worker 可能修改同一文件，或将内部业务逻辑堆到 facade。
- 根因：当前 manifest 只记录 placeholder 和 FFI stub 证据，还不是完整的 rewrite ownership 合同。

#### 解决思路与预期效果

- 解决思路：scaffold generator 按它实际生成的文件和符号写入四类所有权，controller 根据 manifest 授权。
- 预期效果：分类不依赖模型能力和业务名称；同一 scaffold 始终生成相同的所有权合同。

#### 设计方案

1. 扩展 `scaffold-manifest.json` 为四类 path + `frozen_contract_symbols`。
2. 所有 path 必须是项目内相对路径，不允许绝对路径、`..` 或 symlink。
3. 四类 path 两两不相交；可修改的 Rust 文件必须有唯一 owner，未分类文件默认不可写。
4. FFI stub 所在文件可归 facade，但函数签名必须作为符号级冻结合同复核。
5. 如果 generator 无法为某个可写文件确定唯一 owner，PREPARE 直接阻断，不让 agent 猜测，也不降级为两个 worker 共同写。

### Q8-3：新 worker 重复获取历史上下文

#### 问题描述

- 现象：如果 facade worker 重新读取 core worker 的完整 packet、对话、命令输出和实现总结，拆 session 会变成重复上下文消费。
- 影响：总 token 显著增加，主 orchestrator 也可能成为新的长上下文瓶颈。
- 根因：没有把“当前文件状态”与“历史执行过程”分开。

#### 解决思路与预期效果

- 解决思路：当前工作树是实现状态权威；每个 worker 只获取自己的静态 packet、可读范围和必要事实切片。
- 预期效果：单 session 峰值下降；总 token 只增加精简规则和必要代码的重读成本。

#### 设计方案

1. core packet 包含内部 requirement 事实、`core_owned_paths` 和内部实现约束。
2. facade packet 只包含 ABI 函数事实、`facade_owned_paths`、冻结签名和所需内部 Rust API 签名。
3. facade worker 不获取 core worker 的对话、packet、命令输出或自然语言总结。
4. worker 返回主 orchestrator 的内容只包含 `worker_role/revision/status/receipt_path/next_action`，详细证据写入 trace。
5. subagent 无法可靠感知自身 token，停止条件由 contract 配置的 packet 字节、工具输出字节、tool call、check 和 retry 次数强制。

### Q8-4：编译、格式化和 audit 输出放大上下文

#### 问题描述

- 现象：实现 agent 直接运行 cargo/fmt/audit，完整错误、diff 和 warning 进入对话。
- 影响：少数失败命令即可消耗大量上下文，同一诊断在重试中反复出现。
- 根因：验证命令缺少面向 LLM 的有界输出层。

#### 解决思路与预期效果

- 解决思路：验证由 controller/check wrapper 执行，完整 stdout/stderr 写入 trace，模型只收到结构化诊断和有界 tail。
- 预期效果：命令输出不再无上限进入 worker 上下文，退出码和完整日志仍可追溯。

#### 设计方案

1. wrapper 顺序执行 fmt、build 和指定 audit，保留子命令真实退出码。
2. 完整日志写入 `logs/trace/rewrite-check/<role>/<revision>/`，stdout 只输出 contract 配置的最大字节数。
3. 编译诊断按主文件路径归属到 core/facade；audit 诊断按 manifest owner 归属。
4. 无法唯一归属的问题标记 `blocked`，不启动拥有全部写权的通用 repair agent。

### Q8-5：当前 workflow 不支持源码回滚

#### 问题描述

- 现象：`workflowctl` baseline 仅保存 hash，`abort` 只恢复 workflow state，worker 已写入的 Rust 文件仍保留。
- 影响：如果设计为“失败后自动回滚再换 agent”，实现会被迫依赖 Git、备份目录或重生成 scaffold，与已确定安全约束冲突。
- 根因：检测 baseline 与恢复 source snapshot 是两种不同能力，当前只具备前者。

#### 解决思路与预期效果

- 解决思路：不做源码回滚；每类 worker 在自己的 owner paths 上原地向前修复，上游 revision 变化时使下游回执失效并重验。
- 预期效果：不需要危险的 reset/copy/restore；每次修订和重验都有不可变证据。

#### 设计方案

1. core/facade 回执包含 `revision`、`based_on_scaffold_digest`、`based_on_core_revision`、`supersedes`、`changed_files`、`changed_file_hashes` 和 check receipt。
2. core revision 变化后，所有 `based_on_core_revision` 不匹配的 facade/final-check 回执自动失效。
3. 失效不等于删除文件或恢复旧源码；controller 先在当前工作树重验，仅在检查失败时重启相应 owner worker。
4. facade 报告 `missing_core_capability` 时，只能触发 core 新 revision，不能让 facade 越权修改 core。
5. frozen/shared 问题不属于 REWRITE 修复范围，直接阻断并记录上游合同问题；不在活跃 REWRITE 事务中重生成 scaffold。

### Q8-6：worker 自报无法作为可信完成证据

#### 问题描述

- 现象：当前 `core_rewrite_batches.jsonl` 主要依赖 agent 自报 changed files 和 obligations。
- 影响：无法确认 worker 是否越权修改、是否遗漏 stub，也无法在多次 revision 后判断哪份证据仍有效。
- 根因：回执还不是 controller-owned 事务证据。

#### 解决思路与预期效果

- 解决思路：changed files、hash、所有权合规、revision 和 check 状态全由 controller 计算，agent 只返回简短状态。
- 预期效果：最终 gate 能从 scaffold baseline 到最新 core/facade revision 重建整个 REWRITE 过程。

#### 设计方案

1. 每个 worker 开始前由 controller 记录全工作树 hash baseline，结束时计算真实 diff。
2. 真实 changed files 必须全部属于 worker owner paths；frozen/shared 变化立即失败。
3. 回执追加写入，不覆盖历史 revision；最新有效回执通过 revision 链确定。
4. 最终 gate 检查最新 core/facade revision、所有权、冻结符号签名、实现 audit、fmt 和 release build。

## 错误归属与处理

| 错误类型 | 处理方式 | 是否回滚源码 |
|---|---|---|
| 错误主位置属于 `core_owned_paths` | 重试 `IMPLEMENT_CORE` | 否 |
| 错误主位置属于 `facade_owned_paths` | 重试 `WIRE_FACADE` | 否 |
| facade 缺少内部能力 | 新 core revision，然后重验 facade | 否 |
| frozen signature/layout 变化 | 阻断 REWRITE，记录上游合同问题 | 否 |
| shared readonly 被修改 | 当前 worker 失败，不接受回执 | 否；当前工程不具备自动恢复能力 |
| workflow/gate/controller 问题 | 阻断并记录 workbench issue | 否 |
| 无法唯一归属 | 阻断，不启动通用 repair agent | 否 |

shared/frozen 越权变更无法被当前 controller 自动恢复，因此必须通过更小的 worker 写权、开始前权限检查和结束时 diff gate 预防。本方案不宣称已具备不存在的源码恢复能力。

## 上下文和重试预算

下列值是根据本次 trace 给出的建议配置，必须由 workflow contract 读取，不得作为代码常量或项目分支写死。

| 项目 | 建议配置 | 目的 |
|---|---:|---|
| 单 worker packet | 24KB | 限制初始上下文 |
| 单命令返回模型的输出 | 8KB | 防止 cargo/fmt 放大 |
| 单 worker 失败 check | 3 次 | 每次换同职责 session，预算耗尽后阻断 |
| C 源码窗口 | 单次 120 行 | 只处理明确事实缺口 |
| Rust 读取 | 目标符号窗口优先 | 避免反复全文读取 |

模型 token 数不作为 worker 自己执行的 gate，因为当前 OpenCode 不将精确剩余上下文暴露给 subagent。

## 不采用的方案

1. 不依赖 OpenCode 自动 compact 或 agent 自己判断 token。
2. 不按项目业务模块名固定分批。
3. 不让弱模型生成依赖图、拓扑排序、语义聚类或修改调度计划。
4. 不并行启动多个可写源码的 worker。
5. 不新增通用 `INTEGRATION_REPAIR` agent。
6. 不在 worker 之间传递对话或自然语言实现总结。
7. 不使用 Git reset/checkout、`/tmp`、`cp` 备份或重生成 scaffold 回滚 Rust 实现。

## 最终验收目标

1. 不再出现单个 REWRITE subagent 接近 provider 上下文上限的请求。
2. GLM/OpenCode 不参与所有权分类、调度、错误跨 owner 修复或回执生成。
3. 任一 worker 只修改它的 owner paths，冻结路径、共享只读路径和冻结符号保持不变。
4. 修复通过新 revision 向前进行，不使用危险或未授权的源码回滚。
5. 最终 `core_impl_audit`、`cargo fmt --check`、`cargo build --release` 和 REWRITE gate 通过。
6. 不以减少 requirement、跳过模块、改变 ABI 合同或保留 stub 的方式换取上下文下降。
7. 后续 VERIFY/C-CROSS 功能通过率不低于未拆分时的最佳实现。
