# Q3 - 上下文爆炸收敛方案

## 现象

`rust-implementer` subagent（session `ses_0bbac8f84ffeaX1s4nh0Y2lF7Q`）在执行第 23 步时，累积对话上下文超过 GLM-5.1 最大输入长度限制（202,745 tokens）。

本地日志显示的直接原因链条：

1. subagent 从 00:42:26 到 00:48:47 连续运行 23 步，每一步工具调用结果都保留在同一个 session 上下文中。
2. subagent 读取了多个大 C 源文件和测试文件窗口，其中包括 `fdb_kvdb.c`、`fdb_tsdb.c`、`fdb_file.c`、`fdb_utils.c`、`fdb.c` 等。
3. subagent 又写入了多份 Rust 文件，编辑结果、diff、编译错误和验证输出继续进入同一上下文。
4. opencode subagent 没有自动 compact / summarize / resume 机制，历史消息持续累积。

## 判断

这不是单次模型调用异常，也不是“工程文档太多”这个单点导致的。真正问题是：

```text
拆了角色，但没有收紧每个角色的输入边界。
```

以前不拆 subagent 时，主控会把前置阶段、工具调用、日志写入、C 源码读取、Rust 实现和编译修复都累积在主会话里，所以也存在上下文耗尽风险。`design_doc/todolist.md` 里已经记录过 `REWRITE_CORE_MODULES` 因上下文窗口耗尽中断，需要多次 continuation 才完成。

引入 subagent 后，主控上下文确实变短了，但 `rust-implementer` 被绑定到三个重阶段：

```text
GENERATE_RUST_SCAFFOLD
REWRITE_CORE_MODULES
VERIFY_RUST_WITH_C_TESTS
```

它仍然可能连续读取模型、源码、Rust 代码、编译日志和验证结果。于是上下文压力从主控转移到了 `rust-implementer` 的独立 session 中，并且因为没有 compact，爆得更集中。

正确方向不是继续增加 task queue、query 工具或一套新的状态机，而是：

```text
保留现有四个 subagent。
保留现有阶段。
减少每个阶段允许进入上下文的材料。
```

## 当前风险输入

`logs/trace` 是机器证据仓库，不等于 agent 阅读材料。当前高风险输入包括：

- `logs/trace/c_test_model.json`：数千行；
- `logs/trace/rust_api_design.json`：数千行；
- `logs/trace/c_api_model.json`：数千行；
- C 源码和 C 测试文件全文；
- 生成后的 Rust 大文件；
- cargo / linker / cross validate 完整输出；
- 总设计文档和阶段文档全文；
- 历史报告和过程复盘。

这些文件对工具和 gate 有价值，但不应被 subagent 默认全文读取。

## 已确认优化点

### 1. trace 大模型是冷数据

`c_api_model.json`、`c_test_model.json`、`rust_api_design.json` 和 `validation-matrix.json` 继续作为工具和 gate 的事实来源，但 subagent 默认不得全文读取。需要模型事实时，只读取当前阶段直接相关的局部片段。

### 2. 不新增复杂 subagent 任务系统

不新增 task packet、动态 task queue、`rewrite-task-index.json`、`query_context.py` 或一批新的细粒度 subagent。现阶段只收紧现有阶段的输入边界。

### 3. 过程文档用于审计

`design_doc/`、`design_doc/stages/`、`logs/trace/<stage>.md`、`result/output.md` 和 `result/issues/00-summary.md` 不作为 subagent 默认上下文。它们保留给人审计、gate 检查和最终报告引用。

### 4. `rust-implementer` 按现有阶段隔离 session

`rust-implementer` 仍是同一个角色，但三个阶段不要求在同一个对话 session 中连续执行：

```text
GENERATE_RUST_SCAFFOLD
REWRITE_CORE_MODULES
VERIFY_RUST_WITH_C_TESTS
```

每个阶段完成后可退出，下个阶段从文件系统产物恢复上下文。

这个隔离不能导致重复读取 C 工程：

- `GENERATE_RUST_SCAFFOLD` 不读 C 源码；
- `REWRITE_CORE_MODULES` 只有在模型事实不足时，才读取 C 源码局部窗口；
- `VERIFY_RUST_WITH_C_TESTS` 默认只读验证输出、失败 tail 和必要 layout/API 片段。

### 5. 减少执行期过程状态和中间报告

阶段中只保留最小证据：

- `workflow_state.json`；
- `subagent-invocations.jsonl`；
- gate 必需机器产物；
- 短阶段日志；
- 必要 cargo/test tail log。

`result/output.md` 和 `result/issues/00-summary.md` 只在 `REPORT_AND_VERIFY` 阶段统一生成。前置 subagent 不维护最终报告。

### 6. 源码读取是硬边界

`rust-implementer` 默认不读 C 源码。只有 `REWRITE_CORE_MODULES` 阶段可按需读取源码局部窗口，并且必须先说明要验证的模型缺口或实现疑点。

约束：

```text
超过 400 行的 C/Rust 文件禁止全文读取。
必须先用 rg 定位，再读取命中点附近窗口。
单次源码窗口不超过 120 行。
每阶段累计源码窗口不超过 600 行。
```

### 7. subagent 不负责最终报告

subagent 只产出本阶段机器产物、短阶段日志和调用证据。最终报告、最终问题摘要和 `STATUS: SUCCESS` 只由 `REPORT_AND_VERIFY` 阶段统一生成。

### 8. gate 以产物证据为主

`agent-registry.json` 和 `subagent-invocations.jsonl` 保留，但阶段是否通过主要看该阶段机器产物、`workflow_state.json`、验证结果和最终 gate 命令。subagent 调用记录只证明调用或 fallback 发生过，不应反过来制造大量过程状态。

### 9. `c-analyzer` 是唯一默认系统性读取 C 输入的 subagent

`c-analyzer` 可以保持 `READ_C_PROJECT + BUILD_C_MODEL + DESIGN_RUST_API` 一次性执行，因为它的职责就是读一次 C 工程、建模一次、设计一次。

后续 `rust-implementer`、`test-migrator`、`repairer` 不继承系统性读取 C 工程的权限。

### 10. 先改合同，不急着加工具

本轮只改：

```text
INSTRUCTION.md
work/skills/*
design_doc/*
```

不新增查询工具、不新增任务队列、不新增状态机。只有现有测试或 gate 明确要求时，才最小化调整 `gate.py`。

## 落地顺序

第一步，收紧文档合同：

```text
02_02/INSTRUCTION.md

02_02/work/skills/rust-implementer.md
02_02/work/skills/test-migrator.md
02_02/work/skills/repairer.md

02_02/work/skills/flashdb-test-migration/SKILL.md
02_02/work/skills/rust-compile-repair/SKILL.md
design_doc/stages/06-rewrite-core-modules.md
design_doc/stages/06-5-verify-rust-with-c-tests.md
design_doc/参赛工程设计文档.md
```

第二步，运行现有合同测试，删除或更新仍在强制旧长事务、全文读取或过度过程记录的断言。

第三步，只有测试或 gate 仍被旧合同卡住时，才最小化调整 `gate.py`。

如果只做一步，优先做第一步。当前爆点不是缺少工具，而是执行合同允许 `rust-implementer` 在一个 session 里做太多事、读太多东西。
