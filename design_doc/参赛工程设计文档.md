# FlashDB C-to-Rust 参赛工程设计文档

> 当前架构：Q9 简化验证版。历史 controller/packet/worker 设计仅保留在 Q7、Q8 作为复盘材料，不是运行合同。

## 目标

参赛作品在运行时读取平台提供的 FlashDB C 工程，动态生成 `flashDB_rust/`，并用原始 C 测试实际调用 Rust staticlib。工程不提交预置答案，不依赖网络或人工输入，也不修改平台 C 输入。

## 架构

```text
INSTRUCTION.md
    ↓
一个 primary Agent
    ├─ ANALYZE：动态 C/API/ABI/test 事实
    ├─ IMPLEMENT：scaffold 与 Rust 行为实现
    ├─ VERIFY_AND_REPAIR：ABI + 原 C runner 真实执行
    └─ TEST_AND_REPORT：Rust 测试、cargo、报告与最终验证
```

`work/tools/` 只承担确定性事实提取、真实执行或结果检查。Agent 负责基于当前事实选择最小实现和修复；工具不维护任务调度、角色权限、nonce、receipt、packet 或 revision。

## 事实来源

| 问题 | 权威证据 |
|---|---|
| C 输入是否被正确发现 | `input_manifest.json` 与输入基线 |
| API、ABI、测试语义是什么 | C 模型和 Rust API design JSON |
| ABI 是否相同 | 编译器 probe 和 layout check 日志 |
| Rust 是否真正通过 C 测试 | `validation-matrix.json`、compile/link/test 日志 |
| Rust 测试是否逐项迁移 | mapping、placeholder、consistency 结果 |
| 最终结果是否真实 | cargo 结果、unsafe ratio、最终报告 |

状态页、Agent 回复和过程日志不能替代上述证据。

## Agent 与上下文

主 Agent 是默认执行者。subagent 仅用于短、可验证的辅助工作，例如生成测试初稿或修复一组已明确的失败；subagent 不负责阶段推进和成功结论，失败时主 Agent 立即接手。

大 JSON、历史日志和设计文档是冷数据。执行者先定位，再读取局部事实；C-Cross 失败只消费 failure summary、对应日志 tail 和相关代码窗口。会话需要缩短时，从当前文件和机器证据恢复，不传递旧对话历史。

## 验证与报告

ABI mismatch 必须先失败，不能继续假装功能验证。C-Cross 必须构建 Rust staticlib、编译/链接原 C runner 并在项目 owned 的 `logs/trace/c-cross/` 目录运行。失败、未运行和解析失败都如实保留。

无论成功或失败都生成报告。只有全量最终 C-Cross、Rust 测试真实性、cargo、unsafe ratio 与最终 gate 一致通过时，报告可以写 `STATUS: SUCCESS`。
