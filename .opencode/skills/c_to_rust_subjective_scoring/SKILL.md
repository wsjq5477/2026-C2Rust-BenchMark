# c_to_rust_subjective_scoring Skill

## 1. Skill 目标

本 Skill 用于对“C 语言项目转换为 Rust 项目”的参赛作品进行主观评分分析。

本 Skill 只评估主观评分中的：

- 泛用性：20 分
- 规范性：10 分
- 性能：默认给 10 分，不做实际评估

最终输出中文主观评分报告，包含得分、扣分原因、证据位置和优化建议。

## 2. 适用场景

适用于以下比赛或评测场景：

- 参赛作品要求读取 `INSTRUCTION.md`；
- 参赛作品需要将 C/C++ 工程迁移为 Rust 工程；
- 已有单轮或多轮客观评分报告；
- 需要从评委视角评估作品是否具备泛化能力和规范性；
- 性能分暂不分析，直接按满分 10 分计入主观总分。

## 3. 输入要求

执行本 Skill 前，应尽可能提供以下材料：

```text
参赛作品根目录/
├── INSTRUCTION.md
├── work/
│   ├── skills/
│   ├── agents/
│   ├── scripts/
│   └── ...
├── logs/
│   ├── conversion.log
│   ├── build.log
│   ├── test.log
│   └── scoring.log
├── reports/
│   ├── scoring_report.md
│   ├── scoring_report.json
│   └── ...
└── 生成的 Rust 工程/
    ├── Cargo.toml
    ├── src/
    └── tests/
```

如果部分材料缺失，仍然继续评估，但必须在报告中标明“证据不足”并酌情扣分。

## 4. 输出要求

必须输出中文报告：

```text
reports/subjective_score_review.zh.md
```

建议同时输出机器可读 JSON：

```text
reports/subjective_score_review.json
```

Markdown 报告必须包含：

1. 主观评分总览；
2. 泛用性评分；
3. 规范性评分；
4. 性能默认分说明；
5. 关键扣分点；
6. 关键加分点；
7. 优化建议；
8. 证据索引。

## 5. 评分总则

主观评分共 40 分：

```text
主观总分 = 泛用性得分 + 规范性得分 + 性能默认得分
性能默认得分 = 10 分
```

评分时必须遵守：

- 不因为客观测试通过率高就直接给泛用性高分；
- 不因为单个 case 失败就直接给泛用性低分；
- 泛用性重点看是否能迁移同类 C 工程；
- 规范性重点看目录、Skill、Agent、日志、报告、入口是否清晰；
- 所有扣分必须给出证据；
- 没有证据时可以标为“未发现”或“证据不足”，不得编造。

## 6. 泛用性评分规则：20 分

泛用性评估作品是否具备从类似 C 项目迁移到 Rust 项目的通用能力，而不是只针对当前 FlashDB 题目或测试环境硬编码。

### 6.1 泛用性评分维度

| 维度 | 分值 | 评分重点 |
|---|---:|---|
| 任务理解与 INSTRUCTION 解析 | 3 | 是否读取并解析 `INSTRUCTION.md`，是否基于任务说明生成迁移计划 |
| C 工程结构发现能力 | 4 | 是否自动识别源码、头文件、测试入口、构建入口、模块依赖 |
| 跨文件语义迁移能力 | 4 | 是否保持模块关系、API 关系、数据结构语义和错误码语义 |
| 编译/测试自愈闭环 | 4 | 是否根据编译错误、测试失败、日志栈进行定位和修复 |
| 非硬编码与可迁移性 | 3 | 是否避免写死路径、case、返回值、FlashDB 专属逻辑 |
| 产物可扩展性 | 2 | Rust 工程是否模块化，是否便于扩展到同类 C 库 |

总分：20 分。

### 6.2 任务理解与 INSTRUCTION 解析：3 分

加分证据：

- 明确读取 `INSTRUCTION.md`；
- 能从 `INSTRUCTION.md` 提取任务目标、输入工程、输出要求、测试要求；
- 转换日志或报告中能看到基于题目说明形成的迁移计划。

扣分情形：

- 未读取 `INSTRUCTION.md`，直接执行固定流程；
- 只搜索固定文件名；
- 任务目标与 `INSTRUCTION.md` 明显不一致。

建议评分：

```text
0 分：完全无证据表明读取 INSTRUCTION.md
1 分：读取了，但没有形成任务理解
2 分：能基于 INSTRUCTION.md 形成基本流程
3 分：能解析任务约束并影响迁移策略
```

### 6.3 C 工程结构发现能力：4 分

加分证据：

- 自动扫描 C 源文件和头文件；
- 分析 `#include` 关系；
- 识别测试目录和测试入口；
- 识别构建脚本、Makefile、CMake、示例工程或平台脚本；
- 生成源文件清单或依赖图。

扣分情形：

- 写死 `fdb_kvdb.c`、`fdb_tsdb.c` 等固定文件；
- 只迁移少量文件，不解释排除原因；
- 不分析头文件和宏定义；
- 测试入口靠固定路径。

建议评分：

```text
0 分：完全写死源码路径
1 分：能扫描文件，但无依赖分析
2 分：能识别主要源码和头文件
3 分：能识别源码、头文件、测试入口和构建入口
4 分：能形成清晰的工程结构和模块依赖分析
```

### 6.4 跨文件语义迁移能力：4 分

加分证据：

- 保留 C 工程模块边界；
- Rust 模块与 C 源文件或功能模块有映射关系；
- 数据结构转换合理，例如 struct、enum、错误码、状态机、持久化结构；
- API 语义与原 C 版本对齐；
- 对宏、条件编译、平台适配层有处理策略。

扣分情形：

- 所有逻辑堆在单个 `lib.rs`；
- 为过测试伪造返回值；
- 忽略错误码、边界条件、持久化语义；
- 测试能过但实现明显不是 FlashDB 的语义等价迁移；
- 跨文件调用关系被破坏。

建议评分：

```text
0 分：基本没有语义迁移，只是壳或 mock
1 分：迁移了局部函数，但跨文件关系混乱
2 分：核心模块基本迁移，但语义风险较多
3 分：主要 API、数据结构和模块关系较完整
4 分：跨文件语义、错误处理、边界行为都有明确对齐证据
```

### 6.5 编译/测试自愈闭环：4 分

加分证据：

- 有 `cargo check`、`cargo test`、测试报告；
- 能根据 Rust 编译错误修复；
- 能根据测试失败定位到具体模块；
- 每轮修复有日志和原因；
- 修复不是全量重写，而是定点 patch。

扣分情形：

- 编译失败后无修复；
- 测试失败后盲目重写；
- 反复修同一个错误但没有收敛；
- 只为了 case 修改，不解释语义原因；
- 缺少修复日志。

建议评分：

```text
0 分：无自愈闭环
1 分：能尝试修复，但无结构化日志
2 分：有编译修复，测试修复较弱
3 分：编译和测试均有闭环，能定位主要问题
4 分：修复过程清晰、定点、可复盘，能证明通用自愈能力
```

### 6.6 非硬编码与可迁移性：3 分

加分证据：

- 没有写死 `/app/tasks/ICT-4222`、`02_02_赛博三鲜`、固定 judge 目录；
- 没有针对 case 编号写分支；
- 没有直接复用历史 `flashDB_rust`；
- 能通过配置或扫描适配不同源码根目录；
- 有清晰的禁止硬编码说明。

扣分情形：

- 写死题目路径、case 编号、测试名称；
- 对某些测试函数直接返回期望值；
- 检测到已有 `flashDB_rust` 就跳过转换；
- 依赖评测环境中的隐藏路径；
- 使用原始作品测试函数冒充迁移结果。

建议评分：

```text
0 分：严重硬编码
1 分：存在明显 FlashDB 或环境专用逻辑
2 分：少量路径或命名假设，但主体流程可迁移
3 分：基本无硬编码，路径、源码、测试入口均可配置或自动发现
```

### 6.7 产物可扩展性：2 分

加分证据：

- 生成标准 Cargo 工程；
- 模块划分清晰；
- 测试目录清楚；
- 错误类型、配置、存储接口有基本抽象；
- 后续支持其他 C 库迁移时改动较少。

扣分情形：

- Rust 工程不可独立构建；
- 无标准 `Cargo.toml`；
- 模块混乱；
- 大量临时代码、死代码、未使用代码；
- 输出目录不稳定。

建议评分：

```text
0 分：产物不可扩展或不可构建
1 分：Cargo 工程可用，但结构较粗糙
2 分：工程结构清晰，具备继续扩展基础
```

## 7. 规范性评分规则：10 分

规范性评估作品是否符合比赛交付规范、Skill 规范、Agent 规范、日志报告规范和可复现要求。

### 7.1 规范性评分维度

| 维度 | 分值 | 评分重点 |
|---|---:|---|
| 提交目录规范 | 2 | 根目录、`INSTRUCTION.md`、`work/skills`、`work/agents` 等是否规范 |
| Skill / Agent 规范 | 3 | `SKILL.md`、Agent 文件、输入输出、工具约束、流程定义是否清楚 |
| 执行入口与可复现性 | 2 | 是否能按说明启动，是否有明确命令、依赖、环境说明 |
| 日志与报告规范 | 2 | 是否保存完整日志、中文报告、JSON/Markdown 产物索引 |
| 失败处理与边界说明 | 1 | 是否说明失败行为、降级策略、禁止事项 |

总分：10 分。

### 7.2 提交目录规范：2 分

检查项：

- 根目录存在 `INSTRUCTION.md`；
- Skill 目录符合 `work/skills/[skillName]/SKILL.md`；
- Agent 目录符合 `work/agents/[agentName].md`；
- 没有错误结构，例如 `work/skills/SKILL.md`、`work/skill/[skillName]/SKILL.md`；
- 脚本、日志、报告目录清晰。

建议评分：

```text
0 分：目录结构明显不符合提交规范
1 分：主体可识别，但存在不规范目录或产物散乱
2 分：目录结构清晰，符合比赛提交规范
```

### 7.3 Skill / Agent 规范：3 分

检查项：

- Skill 有明确目标；
- Skill 有适用场景；
- Skill 有输入输出定义；
- Skill 有分阶段执行流程；
- Skill 有禁止事项；
- Skill 有日志和报告要求；
- Agent 职责清楚，未与 Skill 混乱；
- 不把大量无关上下文塞入 Skill。

建议评分：

```text
0 分：没有规范 Skill 或 Agent
1 分：有 Skill，但像普通说明文，缺少输入输出和流程约束
2 分：Skill/Agent 基本规范，少量细节不足
3 分：Skill/Agent 职责清晰，流程、约束、产物完整
```

### 7.4 执行入口与可复现性：2 分

检查项：

- 是否说明如何启动；
- 是否说明依赖；
- 是否说明输入路径；
- 是否说明输出路径；
- 是否能在清空上下文后重新运行；
- 是否避免依赖本地绝对路径和历史产物。

建议评分：

```text
0 分：不知道如何运行或严重依赖本地环境
1 分：可以运行，但入口、依赖或路径说明不完整
2 分：入口清晰，依赖明确，结果可复现
```

### 7.5 日志与报告规范：2 分

检查项：

- 转换日志完整；
- 编译日志完整；
- 测试日志完整；
- 评分报告存在；
- 中文报告存在；
- 有 `artifacts_manifest.json` 或类似产物索引；
- 报告能关联源码、日志、测试结果和修复过程。

建议评分：

```text
0 分：日志和报告缺失严重
1 分：有部分日志或报告，但不完整、不易追踪
2 分：日志、报告、产物索引完整，便于复盘
```

### 7.6 失败处理与边界说明：1 分

检查项：

- 说明转换失败如何处理；
- 说明编译失败如何处理；
- 说明测试失败如何处理；
- 说明不得复用旧 Rust 工程；
- 说明不得 fallback 到原始作品；
- 说明无法评估时如何标记。

建议评分：

```text
0 分：缺少失败处理和禁止事项
1 分：失败处理、边界和禁止事项清楚
```

## 8. 性能评分规则：默认 10 分

本 Skill 不实际评估性能。

无论日志中耗时和 Token 情况如何，性能分固定为：

```text
性能得分：10 / 10
```

报告中必须写明：

```text
本次主观评分暂不分析性能，按用户要求默认记为 10 分。该分数不代表真实比赛性能排名。
```

## 9. 执行流程

执行本 Skill 时按以下流程操作。

### 9.1 收集材料

检查以下路径是否存在：

```text
INSTRUCTION.md
work/skills/
work/agents/
work/scripts/
logs/
reports/
Cargo.toml
src/
tests/
```

如果路径不存在，不要中断，记录为证据缺失。

### 9.2 检查硬编码风险

搜索以下模式：

```text
/app/tasks/
ICT-
assistant_judge_
02_02_
赛博三鲜
flashDB_rust
FlashDB_Rust
case10
case_10
case23
case_23
return true
unwrap_or_default
TODO
mock
stub
```

注意：命中不一定扣分，必须结合上下文判断。

重点识别：

- 是否把评测路径写死；
- 是否把题目名称写死；
- 是否把 case 编号写死；
- 是否检测到已有 Rust 工程后跳过转换；
- 是否用 mock/stub 替代真实语义。

### 9.3 检查泛用性证据

阅读以下材料：

- `INSTRUCTION.md`；
- 转换日志；
- 迁移计划；
- 生成 Rust 工程结构；
- 测试报告；
- 修复日志；
- Agent/Skill 说明。

为每个泛用性维度给出：

```text
得分：x / 满分
证据：...
扣分原因：...
优化建议：...
```

### 9.4 检查规范性证据

阅读以下材料：

- 根目录结构；
- `work/skills/*/SKILL.md`；
- `work/agents/*.md`；
- 运行说明；
- 日志目录；
- 报告目录；
- 产物索引。

为每个规范性维度给出：

```text
得分：x / 满分
证据：...
扣分原因：...
优化建议：...
```

### 9.5 汇总主观分

计算：

```text
泛用性得分 = 六个泛用性维度之和，满分 20
规范性得分 = 五个规范性维度之和，满分 10
性能得分 = 10
主观总分 = 泛用性得分 + 规范性得分 + 10
```

## 12. 评分注意事项

### 12.1 不要过度依赖客观分

客观测试通过率可以作为参考，但不能替代主观判断。

例如：

- 通过率高，但明显硬编码 case，泛用性应扣分；
- 通过率一般，但迁移流程通用、结构规范，可以保留一定泛用性分；
- 某次 judge 失败，不等于规范性一定差，需要看失败原因。

### 12.2 必须区分“FlashDB 专用知识”和“嵌入式 C 库通用能力”

允许作品理解 FlashDB 的业务语义，但不允许只为 FlashDB 当前测试环境写死逻辑。

合理情况：

```text
识别 FlashDB 的 KVDB、TSDB、port、utils 等模块，并映射为 Rust 模块。
```

不合理情况：

```text
如果 case_id == 10，则返回固定值。
```

### 12.3 硬编码命中要结合上下文

搜索到 `flashDB_rust` 不一定扣分，因为它可能只是输出目录名称。

需要扣分的是：

- 发现已有 `flashDB_rust` 后直接复用；
- 路径写死导致换题目不可运行；
- case 编号写死；
- 测试结果写死；
- fallback 到原始 C 工程或原始测试函数。

### 12.4 没有证据时要明确写“证据不足”

禁止凭空假设作品有某能力。

示例：

```text
未发现源码依赖分析日志，因此“C 工程结构发现能力”暂按 2/4 评分。
```

## 13. 最终输出口径

最终报告应使用客观、评审式中文，不要使用夸张表述。

推荐结论格式：

```text
本作品在规范性方面表现较好，目录结构和 Skill 入口较清晰；泛用性方面具备基本的 C 工程扫描和 Rust 工程生成能力，但当前证据仍偏 FlashDB 场景，缺少明确的跨项目抽象和依赖分析报告。性能部分按要求默认给 10 分。因此本次主观评分预估为 xx/40。
```
